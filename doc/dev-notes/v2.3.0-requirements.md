## Release Story

v2.2.0 closed the **MNE-C CLI gap** (82/82 tools), introduced the **MNA/MNX
project format**, the **`ml`/`sts`/`mna` libraries**, **CMNE**, **sparse
inverses**, **directed connectivity**, and **MNX-as-MNE-Scan-storage**. It
also landed the disp3D **`ElectrodeObject`** and **`SliceObject`**
renderables — but neither is exposed in any GUI yet. The remaining
cross-ecosystem deltas now sit in five buckets:

1. **Multimodal visualization in MNE Inspect** is the single largest
   user-facing gap. The disp3D primitives for sEEG depth shafts, ECoG
   grids/strips, and MRI slice planes shipped in v2.2.0 but are not loaded,
   picked, or coloured by data anywhere in the GUI. Without this,
   intracranial users — the fastest-growing mne-cpp audience — cannot use
   the tool.
2. **MNE Scan** is feature-complete in libraries but not yet *production-ready*
   as a turn-key real-time platform — packaging, plugin coverage, RT inverse
   wiring, and recording/replay UX still need work.
3. **mne_analyze** exposes ~70 % of the legacy MNE-C `mne_analyze` workflow
   but is missing the **cortical-surface source-estimation workflow** that
   most clinicians actually open the tool for (load surface → load STC →
   pick vertex → see timecourse).
4. **mne_analyze_studio** ships its skill/agent host but needs a production
   workflow showcase and tighter MNA integration.
5. **MNE-Python** still has roughly a dozen high-priority algorithms we don't
   ship (TF-MxNE, standard montages, Maxwell movement compensation,
   multitaper TFR, Picard ICA, surface Laplacian, depth priors, full
   `SourceMorph`, `simulate_raw/_stc/_evoked`), and an additional ~30
   medium-priority gaps in covariance estimation, preprocessing, source
   containers, decoding/spatial-filters, I/O export, connectivity, and
   reporting that together prevent full feature parity.

v2.3.0 attacks all five buckets in parallel. The unifying theme: **stand
up MNE Inspect as the multimodal viewer for intracranial + scalp + MRI +
source data, make MNE Scan production-ready, land the cortical workflow in
mne_analyze, and close the remaining MNE-Python algorithm parity items.**

## TASK 1 — mne-mna Python Package for MNA/MNX Interoperability

**Goal:** Create a Python package (`mne-mna`) that reads and writes `.mna`
(JSON) and `.mnx` (CBOR) files, integrating with MNE-Python workflows. This
enables Python-based researchers to consume and produce MNA project files,
bridging the MNE-CPP and MNE-Python ecosystems.

### Rationale for v2.3.0 (not v2.2.0)

- The MNA/MNX format should be published and citable (v2.2.0 TASK 17) before
  approaching the MNE-Python community for adoption.
- Coordination with MNE-Python maintainers is needed for API design review
  and potential joint publication.
- One more release cycle allows the format to stabilize based on internal
  usage and feedback.

### Package scope

#### Core I/O

| Function | Description |
|----------|-------------|
| `mne_mna.read_project(path)` | Read `.mna` (JSON) or `.mnx` (CBOR) → `MnaProject` object |
| `mne_mna.write_project(project, path)` | Write `MnaProject` → `.mna` or `.mnx` (auto-detected by extension) |
| `mne_mna.mna_to_mnx(in_path, out_path)` | Convert JSON ↔ CBOR |

#### Data model (Python classes mirroring C++)

| Class | Maps to C++ |
|-------|-------------|
| `MnaProject` | `MNALIB::MnaProject` |
| `MnaGraph` | `MNALIB::MnaGraph` |
| `MnaNode` | `MNALIB::MnaNode` |
| `MnaPort` | `MNALIB::MnaPort` |
| `MnaSubject` | `MNALIB::MnaSubject` |
| `MnaSession` | `MNALIB::MnaSession` |
| `MnaRecording` | `MNALIB::MnaRecording` |
| `MnaFileRef` | `MNALIB::MnaFileRef` |
| `MnaScript` | `MNALIB::MnaScript` |
| `MnaVerification` | `MNALIB::MnaVerification` |
| `MnaParamTree` | `MNALIB::MnaParamTree` |
| `MnaParamBinding` | `MNALIB::MnaParamBinding` |
| `MnaOpSchema` | `MNALIB::MnaOpSchema` |

Enums: `MnaDataKind`, `MnaFileRole`, `MnaNodeExecMode`, `MnaPortDir` as
Python `enum.Enum` classes with identical string mappings.

#### MNE-Python integration

| Feature | Description |
|---------|-------------|
| `project.to_raw(node_id)` | Extract `mne.io.Raw` from a project's file refs |
| `project.to_epochs(node_id)` | Extract `mne.Epochs` |
| `project.to_source_estimate(node_id)` | Extract `mne.SourceEstimate` |
| `project.add_raw(raw, subject, session, recording)` | Add an `mne.io.Raw` as a file ref |
| `MnaGraph.to_networkx()` | Export DAG as a NetworkX graph for visualization |
| `MnaGraph.from_networkx(G)` | Import DAG from NetworkX |

#### BIDS integration

| Feature | Description |
|---------|-------------|
| `mne_mna.from_bids(bids_path)` | Create `MnaProject` from an existing BIDS dataset (using `mne_bids`) |
| `mne_mna.to_bids(project, bids_root)` | Export file refs into BIDS directory structure |

### Dependencies

- `cbor2` — CBOR encoding/decoding (for `.mnx`)
- `mne` — MNE-Python (optional; required for `to_raw()` etc.)
- `mne_bids` — MNE-BIDS (optional; required for BIDS integration)

### Package structure

```
mne-mna/
  pyproject.toml
  README.md
  LICENSE
  src/
    mne_mna/
      __init__.py
      _project.py      # MnaProject, MnaSubject, MnaSession, etc.
      _graph.py         # MnaGraph, MnaNode, MnaPort
      _io.py            # read_project, write_project, mna_to_mnx
      _types.py         # Enums (MnaDataKind, MnaFileRole, etc.)
      _script.py        # MnaScript
      _verification.py  # MnaVerification, MnaProvenance
      _params.py        # MnaParamTree, MnaParamBinding
      _schema.py        # MnaOpSchema
      _mne_bridge.py    # to_raw, to_epochs, etc. (optional mne dep)
      _bids_bridge.py   # from_bids, to_bids (optional mne_bids dep)
  tests/
    test_io.py
    test_roundtrip.py
    test_mne_bridge.py
    test_bids_bridge.py
    conftest.py
```

### Cross-validation with MNE-CPP

- Round-trip test: C++ writes `.mna` → Python reads → Python writes → C++
  reads → verify identical.
- Same for `.mnx` (CBOR).
- Use `resources/examples/` MNA files as shared test fixtures.

### Publication coordination

- Contact MNE-Python core team (Gramfort, Larson, Engemann) after v2.2.0
  release with the published format paper DOI.
- Propose `mne-mna` as a companion package (similar to `mne-bids`,
  `mne-connectivity`, `mne-icalabel`).
- Joint publication opportunity: MNA/MNX as a reproducible analysis container
  format for the MNE ecosystem.

### Acceptance criteria

1. `pip install mne-mna` installs cleanly on Python 3.10+.
2. Round-trip `.mna` read/write preserves all fields.
3. Round-trip `.mnx` read/write preserves all fields including embedded binary data.
4. `MnaProject` ↔ MNE-Python object conversion works for Raw, Epochs, and
   SourceEstimate.
5. `from_bids()` correctly imports a BIDS dataset into MNA project structure.
6. All enum string mappings match C++ implementation exactly.
7. CI cross-validates Python ↔ C++ round-trips on every commit.

## TASK 2 — NeuralSet Interoperability & Unified Event Model

**Carried over from v2.2.0 TASK 33** (deferred 26 April 2026). Meta released
NeuralSet in April 2026; full interop is a substantial late-binding feature
that should not block v2.2.0.

### Scope summary

- **`MnaEventTable`** — flat event table mirroring NeuralSet's Event model
  (`type`, `start`, `duration`, `timeline`, `extra`); JSON + CBOR I/O; CSV /
  Parquet export; embedded in `MnaRecording`.
- **Extended `MnaDataKind` / `MnaFileRole`** — add `Fmri`, `Fnirs`, `Spikes`,
  `Audio`, `Video`, `Image`, `Embedding`, `EventTable`.
- **Op registry tags** — add `tags` array to `MnaOpSchema`; categorise built-in
  ops as `loader` / `transform` / `extractor` / `external`.
- **`segment_events` op** — declarative event-locked segmenter (NeuralSet
  `Segmenter` parity).
- **`MnaExporter`** — BIDS-aligned event CSV export, BIDS layout export,
  NeuralSet `Chain.from_json()` export.
- **`MnaImporter`** — BIDS dataset → `MnaProject` and `*_events.tsv` →
  `MnaEventTable` importers.
- **BIDS-aligned timelines** — add `task` and `run` fields to `MnaRecording`;
  timeline string `"sub-<id>_ses-<id>_task-<task>_run-<run>"`.

### Reference

Full design specification lives in
[v2.2.0-requirements.md § TASK 33](v2.2.0-requirements.md#task-33--neuralset-interoperability--unified-event-model--deferred-to-v230)
— the design carries over verbatim as the v2.3.0 baseline.

### Dependencies

- v2.2.0 must be released first (NeuralSet interop should target a stable
  MNA/MNX format).
- Coordination with NeuralSet maintainers (Meta / FAIR — Jean-Rémi King team).

### Acceptance criteria

1. `MnaProject` round-trips through NeuralSet's BIDS layout without data loss.
2. NeuralSet `Study.load()` consumes `MnaExporter::exportBidsLayout()` output.
3. `MnaEventTable::writeCsv()` output is consumed by NeuralSet's events frame.
4. `segment_events` op produces `Epochs` matching MNE-Python `Epochs(...)` for
   the same trigger/window definition.
5. CI cross-validates a NeuralSet → MNA → NeuralSet round-trip.

so the GUI supports the canonical *load surface → load STC → pick vertex →

legacy MNE-C `mne_analyze`.

This task is the user-visible counterpart to v2.2.0's library work
(MxNE, Gamma-MAP, CMNE, source-space cluster permutation tests): the
algorithms exist; the GUI now needs to expose them.

  `lh.inflated`, `rh.inflated`, `lh.pial`, `rh.pial`, `lh.white`,
  `rh.white` via existing `FsSurface`.
- 3D rendering re-uses `BrainSurface` QRhi pipeline.

  (`inflated | pial | white`) in the controls dock.
- Curvature shading (sulci/gyri) using `FsSurface::curvature()`.

### Sub-task 4.2 — STC overlay loading & rendering

- "Load Source Estimate…" menu action; reads `.stc` (already in `MNE`
  library) into a new `SourceEstimateModel`.

  (mirroring MNE-C `mne_analyze` overlay controls).
- Time slider drives current sample; "Play" button animates through time
  points.

  v2.2.0).

### Sub-task 4.3 — Vertex picking with timecourse readout

- 3D pick on the cortical surface returns the nearest vertex.
- New "Source Time Course" dock displays the full STC time series at the
  picked vertex with a vertical cursor synced to the time slider.
- Multiple picks accumulate in a "Time Course Manager" list (name,
  colour, hemisphere, vertex index, save/export to CSV).

### Sub-task 4.4 — Source estimation in the GUI

New "Compute Source Estimate" dialog driven by an existing inverse
operator (or computed on the fly from a forward + cov + evoked already
loaded in the BIDS tree):

- Method dropdown: `MNE | dSPM | sLORETA | eLORETA | MxNE | Gamma-MAP |
  CMNE`.
- SNR / λ² control for linear methods; sparsity hyper-parameter for
  sparse methods; ONNX model path for CMNE.
- Output STC populates the `SourceEstimateModel` so it overlays on the
  loaded surface immediately.

### Sub-task 4.5 — Label management

New "Labels" dock:

- Load a single `.label` or batch-load `.annot` (Desikan-Killiany,
  Destrieux); reuses `FsAtlasLookup` from v2.2.0.
- Toggle per-label visibility on the surface.
- Right-click label → "Extract Time Course" with collapse method
  (`mean | max | L2 | L1 | mean+sign-flip`); appends to the Time Course
  Manager.
- "Create Label from Marked Vertices…" — interactive vertex marking +
  flood-fill, save as `.label`.

### Sub-task 4.6 — Field mapping (stretch goal)

If time permits in the cycle:

## TASK 11 — Coregistration Helper Application (Pillar G)

**Goal:** Ship a small, focused standalone application — `mne_coreg` — that
guides the operator step by step through the full EEG/MEG coregistration
workflow using a **Polhemus FASTRAK** pen. The app presents a live 3-D
view of the head model, digitised points, and optional EEG-cap montage,
and produces the **registration matrix** (MRI ↔ Head ↔ Device) needed
for forward modelling and source localisation.

### Rationale

Source localisation requires the precise geometric chain
**MRI → head → sensor/EEG-cap**. Today, operators must manually juggle
FIFF dig files, BEM surfaces, coordinate transforms, and ICP alignment in
the existing `mne_analyze` coregistration plugin — a workflow that was
never designed as a guided standalone tool. A dedicated helper application
with a wizard-style UI, real-time 3-D feedback, and Polhemus FASTRAK
integration eliminates the most error-prone step in the clinical EEG/MEG
pipeline.

### Coordinate frames involved

| Frame | Origin | Used for |
|-------|--------|----------|
| **MRI** (surface RAS) | Centre of the MRI volume | BEM head surface, cortical surface, anatomical landmarks |
| **Head** | Defined by LPA / Nasion / RPA fiducials | Common reference for sensors and digitised head shape |
| **Device** | Polhemus FASTRAK transmitter | Raw digitised point cloud |
| **EEG cap** (head) | Electrode montage positions | EEG channel locations on the scalp (already in head coords after registration) |

### Existing infrastructure (reused)

| Component | Location | Reuse |
|-----------|----------|-------|
| `MNEBem` / `MNEBemSurface` | `src/libraries/mne/` | 3-D head surface loading |
| `FiffDigPointSet` / `FiffDigPoint` | `src/libraries/fiff/` | Digitiser point storage |
| `FiffCoordTrans` | `src/libraries/fiff/` | Transformation matrix I/O |
| `fitMatchedPoints()` | `src/libraries/mne/mne_icp.h` | Fiducial alignment (weighted SVD) |
| `performIcp()` | `src/libraries/mne/mne_icp.h` | ICP refinement |
| `MNEProjectToSurface` | `src/libraries/mne/` | Closest-point projection onto BEM |
| `discard3DPointOutliers()` | `src/libraries/mne/mne_icp.h` | Outlier removal before ICP |
| `CoregSettingsView` | `src/libraries/disp/viewers/` | Existing parameter widgets (partial reuse) |

### Sub-task 11.1 — Polhemus FASTRAK serial driver

New class `PolhemusConnection` (in the application, not a library — keep
scope narrow):

- Opens a serial port (`QSerialPort`) to the Polhemus FASTRAK
  receiver.
- Configures the unit to output Cartesian coordinates in centimetres
  (ASCII mode, continuous or single-point).
- Emits `pointReceived(int station, QVector3D position, QQuaternion
  orientation)` for each stylus trigger.
- Supports both **single-shot** mode (operator presses stylus button to
  record one point) and **continuous stream** mode (head-shape scanning
  at the device's native rate, typically 120 Hz).
- Auto-detects available serial ports; shows a connection dialog with
  port / baud-rate selector and a "Test" button that reads one point and
  displays it.
- Robust error handling: timeout, cable disconnect, garbled data.

### Sub-task 11.2 — 3-D viewport

`Coreg3DView` — a `QRhiWidget`-based 3-D viewport embedded in the main
window:

- Renders the head BEM surface (`BrainSurface` or a lightweight mesh
  renderable) with translucent shading so digitised points underneath
  are visible.
- Overlays digitised points as coloured spheres:
  - **Red** — cardinal fiducials (LPA, Nasion, RPA).
  - **Orange** — additional anatomical landmarks (Inion, optional extras).
  - **Blue** — head shape points (HSP).
  - **Green** — HPI coils (if present).
  - **Yellow** — EEG electrode positions.
- Overlays the EEG-cap montage (standard or custom) as labelled spheres
  on the scalp, projected onto the BEM surface if needed.
- Displays the current RMSE of the ICP fit as a status badge.
- Orbit / pan / zoom interaction (reuse `disp3D` camera).
- Light-weight — no full `MultimodalScene` needed; minimal renderables.

### Sub-task 11.3 — Wizard workflow controller

`CoregWizard` — a `QStackedWidget` driving the operator through the
following guided steps:

#### Step 0 — Setup

- **Load head model**: file picker for the BEM `.fif` (inner-skull or
  head surface from FreeSurfer) or a generic 3-D head mesh (`.obj`,
  `.stl`). Display the mesh in the 3-D view.
- **Load montage (optional)**: choose a standard montage (10-20 / 10-10 /
  10-05 / biosemi-* / easycap-*) from the dropdown (populated by
  `FiffStandardMontage` from TASK 6.2), or load a custom `.elc` / `.sfp` /
  `.tsv` file. Montage positions are shown as labelled yellow spheres on
  the head model.
- **Connect Polhemus**: serial port selection, baud rate, test ping.

#### Step 1 — Fiducial digitisation

Sequential guided acquisition of anatomical landmark points using the
Polhemus stylus. The UI highlights which point to digitise, shows an
anatomical diagram, and waits for the stylus trigger:

| # | Point | FIFF constant | Description shown to operator |
|---|-------|---------------|-------------------------------|
| 1 | **LPA** (left preauricular) | `FIFFV_POINT_LPA` | "Place the stylus on the left preauricular point (just in front of the left ear canal)." |
| 2 | **Nasion** | `FIFFV_POINT_NASION` | "Place the stylus on the nasion (the bridge of the nose between the eyes)." |
| 3 | **RPA** (right preauricular) | `FIFFV_POINT_RPA` | "Place the stylus on the right preauricular point (just in front of the right ear canal)." |
| 4 | **Inion** (optional) | `FIFFV_POINT_EXTRA` | "Place the stylus on the inion (the bump at the back of the skull). Press Skip if not needed." |

Each point:
- Appears in the 3-D view as a red sphere with label.
- Can be **re-digitised** ("Redo") if the operator is not satisfied.
- Is stored as a `FiffDigPoint` with `kind = FIFFV_POINT_CARDINAL`.

After all fiducials are captured, the wizard computes the **Device →
Head** transform from the three cardinal points using `fitMatchedPoints()`.

#### Step 2 — Head shape digitisation

- Switch Polhemus to continuous-stream mode.
- Operator sweeps the stylus over the scalp; each received point is
  immediately shown in the 3-D view as a blue sphere.
- Real-time point counter display ("142 head shape points").
- "Pause / Resume" button for repositioning.
- "Done" button when sufficient coverage is achieved.
- All points stored as `FiffDigPoint` with `kind = FIFFV_POINT_EXTRA`.

#### Step 3 — EEG cap digitisation (optional, skip if no cap)

- If a montage was loaded in Step 0, the wizard shows the first
  electrode label ("Fp1") and waits for a stylus trigger.
- Sequential electrode-by-electrode guided acquisition: the current
  electrode is highlighted on the 3-D head model and in the montage
  diagram.
- Each digitised position is stored as `FiffDigPoint` with `kind =
  FIFFV_POINT_EEG`.
- "Skip electrode" button for electrodes that cannot be reached.
- After all electrodes: compare digitised positions to the ideal montage
  and report per-electrode deviation (mm) — flags any electrode > 10 mm
  off for re-digitisation.

#### Step 4 — HPI coil digitisation (optional, skip if no HPI)

- If MEG system HPI coils are present, digitise each coil location.
- Stored as `FiffDigPoint` with `kind = FIFFV_POINT_HPI`.

#### Step 5 — Registration computation

Automated computation with live progress:

1. **Device → Head** transform — already computed from fiducials in
   Step 1.
2. **Head → MRI** transform — ICP alignment:
   - Initial guess: align MRI fiducials (from the BEM or a `.fif`
     fiducials file) to the digitised Head fiducials using
     `fitMatchedPoints()`.
   - Refinement: `performIcp()` using head-shape points projected onto
     the BEM surface via `MNEProjectToSurface`.
   - Outlier rejection with `discard3DPointOutliers()` (max distance
     threshold configurable, default 20 mm).
   - Optional scaling (uniform or 3-axis) toggle.
   - RMSE display, convergence plot.
3. **Complete chain**: `Device → Head → MRI` composed transform.
4. If an EEG cap was digitised: the **EEG electrode positions in MRI
   coordinates** are computed by applying the full transform chain.

The 3-D view now shows the head model, all digitised points transformed
into MRI space, and the EEG cap — the operator visually verifies the
alignment.

#### Step 6 — Manual adjustment (optional)

- Six sliders for manual rotation (Rx, Ry, Rz) and translation
  (Tx, Ty, Tz) adjustments on top of the ICP result.
- Scaling sliders (uniform or 3-axis).
- "Re-fit ICP" button to restart ICP from the current manual position.
- Live RMSE readout updates as the operator adjusts.

#### Step 7 — Save & export

- **Save transformation**: write `FiffCoordTrans` (Head → MRI) to a
  `-trans.fif` file.
- **Save digitisation**: write all `FiffDigPoint`s to a FIFF file or
  `.hpts` (compatible with `mne_convert_dig_data` output).
- **Save EEG positions in MRI coords** (optional): CSV or `.elc` with
  electrode names and MRI-space positions — directly usable by forward
  modelling.
- **Save to MNA project** (optional): store the transform and dig data
  as `MnaFileRef` entries in an open `.mna` project.
- Summary report: text log of all computed transforms, RMSE, number of
  points, warnings.

## TASK 12 — Covariance Estimation Parity & Inverse Convenience API (Pillar D)

**Goal:** Close the covariance estimation gap and add missing inverse /
beamformer convenience functions so the full MNE-Python inverse workflow
(`compute_covariance → make_inverse_operator → apply_inverse_epochs → 
source PSD`) runs natively in MNE-CPP without lossy workarounds.

### Rationale

MNE-Python offers 8 covariance estimation methods with built-in
cross-validation (`method='auto'`). MNE-CPP currently supports only
empirical covariance (in `fiff/`) and Ledoit-Wolf shrinkage (in `sts/`)
— and they are not integrated into the inverse pipeline. This means users
cannot select the best regularised covariance for their data, which
directly impacts inverse solution quality. Additionally, MNE-Python's
`apply_inverse_epochs()`, beamformer epoch/CSD variants, and source-space
spectral functions have no MNE-CPP counterpart — users cannot compute
per-epoch source estimates or source-space power spectral density.

### Sub-task 12.1 — Covariance estimation methods

Extend `FiffCov` / new `CovEstimator` class in `src/libraries/mne/` (or
`src/libraries/dsp/`) with the following estimators matching MNE-Python's
`compute_covariance()`:

| Method | Algorithm | Notes |
|--------|-----------|-------|
| `empirical` | Sample covariance | Already exists |
| `diagonal_fixed` | Fixed diagonal regularisation | New |
| `shrunk` | Ledoit-Wolf shrinkage | Move from `sts` into cov pipeline |
| `oas` | Oracle Approximating Shrinkage | New |
| `factor_analysis` | Factor Analysis (EM) | New |
| `pca` | PCA-based rank-reduced covariance | New |
| `auto` | Cross-validated selection (log-likelihood) | New — runs all methods, picks best via 3-fold CV |

Additional covariance functions:
- `regularize(cov, info, ...)` — apply diagonal / magnitude regularisation
  to an existing `FiffCov`, matching MNE-Python's `mne.cov.regularize()`.
- `computeWhitener(cov, info, ...)` → whitening matrix; matches
  `mne.cov.compute_whitener()`.
- `prepareNoiseCov(cov, info, ...)` — project, regularise, rank-reduce
  covariance for inverse operator creation.
- `computeRawCovariance(raw, ...)` — estimate covariance from continuous
  data with overlapping windows; matches
  `mne.compute_raw_covariance()`.

### Sub-task 12.2 — Inverse convenience functions

Add to `InvMinimumNorm` or as free functions in `src/libraries/inv/`:

| Function | Description |
|----------|-------------|
| `applyInverseEpochs(epochs, inverse, lambda2, method)` | Apply inverse to each epoch → `QList<InvSourceEstimate>` |
| `applyInverseRaw(raw, inverse, lambda2, method, ...)` | Apply inverse to raw data blocks → continuous `InvSourceEstimate` |
| `applyInverseCov(cov, inverse, ...)` | Apply inverse to covariance → spatial power map (`InvSourceEstimate`) |
| `applyInverseTfr(tfr, inverse, ...)` | Apply inverse to TFR epochs (time-frequency source estimate) |
| `estimateSnr(evoked, inverse)` | Estimate SNR from inverse operator |

### Sub-task 12.3 — Beamformer convenience functions

Add to `InvLCMV` / `InvDICS`:

| Function | Description |
|----------|-------------|
| `applyLCMVEpochs(epochs, beamformer)` | Apply LCMV to each epoch → per-epoch source estimates |
| `applyDICSEpochs(epochs, beamformer)` | Apply DICS to each epoch |
| `makeLCMVResolutionMatrix(fwd, info, cov)` | LCMV resolution matrix (PSF/CTF analysis for beamformers) |

### Sub-task 12.4 — Source-space spectral analysis

New functions in `src/libraries/inv/` or `src/libraries/dsp/`:

| Function | Description |
|----------|-------------|
| `computeSourcePsd(raw, inverse, fmin, fmax, method)` | Compute PSD in source space (apply inverse to each frequency band) |
| `computeSourcePsdEpochs(epochs, inverse, ...)` | PSD from epoched data |

## TASK 14 — Source Estimate Types, Label Operations & Source Spectral Analysis (Pillar D)

**Goal:** Complete the source-space data container hierarchy and
label-based analysis tools to match MNE-Python's rich source-estimate API.

### Rationale

MNE-Python offers 6 source-estimate types (`SourceEstimate`,
`VectorSourceEstimate`, `VolSourceEstimate`, `VolVectorSourceEstimate`,
`MixedSourceEstimate`, `MixedVectorSourceEstimate`). MNE-CPP has a single
`InvSourceEstimate` with a type enum. The lack of proper vector and
volume containers means volumetric inverse solutions (LCMV on volume
source space, volume MxNE) and vector-valued estimates (free-orientation
inverse) cannot be represented, stored, or plotted correctly.

Similarly, MNE-Python's `extract_label_time_course()` with its 5
aggregation modes (`mean`, `mean_flip`, `pca_flip`, `max`, `auto`) is the
standard way to reduce vertex-level data to ROI-level time courses for
connectivity and statistics. MNE-CPP has no library-level equivalent.

### Sub-task 14.1 — Source estimate type hierarchy

Extend `InvSourceEstimate` in `src/libraries/inv/`:

```
InvSourceEstimate          (existing — surface, scalar)
├── InvVectorSourceEstimate    (surface, 3-component vector per vertex)
├── InvVolSourceEstimate       (volume grid, scalar)
├── InvVolVectorSourceEstimate (volume grid, 3-component vector)
├── InvMixedSourceEstimate     (surface + volume, scalar)
└── InvMixedVectorSourceEstimate (surface + volume, vector)
```

Each subclass:
- Stores data with correct dimensionality (scalar: `vertices × times`;
  vector: `3*vertices × times` or `vertices × 3 × times`).
- Implements I/O:
  - `.stc` / `.stc.lh` / `.stc.rh` for surface (existing).
  - `.w` for surface (single time point).
  - `.nii` / `.nii.gz` for volume (via `MriVolData`).
  - `.h5` for all types (HDF5 container with metadata).
- Implements `magnitude()` — vector → scalar via L2 norm.
- Implements `normalComponent(forward)` — extract component normal to
  cortical surface.

### Sub-task 14.2 — Label time course extraction

New function `extractLabelTimeCourse()` in `src/libraries/inv/` or
`src/libraries/mne/`:

```cpp
MatrixXd extractLabelTimeCourse(
    const InvSourceEstimate& stc,
    const QList<FsLabel>& labels,
    const MNESourceSpaces& src,
    const QString& mode = "mean_flip",  // "mean", "mean_flip", "pca_flip", "max", "auto"
    bool allowEmpty = false
);
```

- Returns `n_labels × n_times` matrix.
- `mean` — simple average of vertices in label.
- `mean_flip` — flip vertex signs so they align with the dominant
  orientation (via `label_sign_flip()`).
- `pca_flip` — PCA of label data, flip to align with dominant sign.
- `max` — maximum absolute value in label at each time point.
- `auto` — `mean_flip` for scalar, `mean` for vector.

Helper: `labelSignFlip(label, src)` — compute sign-flip vector from
surface normals.

New functions in `src/libraries/inv/` or `src/libraries/mne/`:

| Function | Description |
|----------|-------------|
| `stcToLabel(stc, src, threshold, ...)` | Threshold STC values → list of `FsLabel`s (one per contiguous supra-threshold cluster per hemisphere) |
| `labelsToStc(labels, values, src, ...)` | Create STC from label-level values (broadcast to all vertices in each label) |
| `stcNearSensors(stc, info, trans, ...)` | Project STC values to nearest sensor → pseudo-evoked for topo plotting |

### Sub-task 14.4 — Label growth & splitting

New functions in `src/libraries/fs/` or `src/libraries/mne/`:

| Function | Description |
|----------|-------------|
| `growLabels(src, seeds, extents, ...)` | Grow labels from seed vertices on the cortical surface (geodesic distance) |
| `splitLabel(label, src, parts)` | Split a label into N sub-labels (e.g. anterior/posterior) |

Extend `disp3D` with a `VolSourceEstimateObject` renderable:
