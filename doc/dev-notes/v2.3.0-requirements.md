Release target: v2.3.0
Codename: **Real-Time Ready · Cortical Studio · Python Bridge**
Branch: staging (all work happens on staging — never push directly to main)

## Release Story

v2.2.0 closed the **MNE-C CLI gap** (82/82 tools), introduced the **MNA/MNX
project format**, the **`ml`/`sts`/`mna` libraries**, **CMNE**, **sparse
inverses**, **directed connectivity**, and **MNX-as-MNE-Scan-storage**. It
also landed the disp3D **`ElectrodeObject`** and **`SliceObject`**
renderables — but neither is exposed in any GUI yet. The remaining
cross-ecosystem deltas now sit in five buckets:

1. **Multimodal visualization in MNE Inspect** is the single largest
   user-facing gap. The disp3D primitives for sEEG depth shafts, ECoG
   grids/strips, and MRI slice planes shipped in v2.2.0 but are not loaded,
   picked, or coloured by data anywhere in the GUI. Without this,
   intracranial users — the fastest-growing mne-cpp audience — cannot use
   the tool.
2. **MNE Scan** is feature-complete in libraries but not yet *production-ready*
   as a turn-key real-time platform — packaging, plugin coverage, RT inverse
   wiring, and recording/replay UX still need work.
3. **mne_analyze** exposes ~70 % of the legacy MNE-C `mne_analyze` workflow
   but is missing the **cortical-surface source-estimation workflow** that
   most clinicians actually open the tool for (load surface → load STC →
   pick vertex → see timecourse).
4. **mne_analyze_studio** ships its skill/agent host but needs a production
   workflow showcase and tighter MNA integration.
5. **MNE-Python** still has roughly a dozen high-priority algorithms we don't
   ship (TF-MxNE, standard montages, Maxwell movement compensation,
   multitaper TFR, Picard ICA, surface Laplacian, depth priors, full
   `SourceMorph`, `simulate_raw/_stc/_evoked`).

v2.3.0 attacks all five buckets in parallel. The unifying theme: **stand
up MNE Inspect as the multimodal viewer for intracranial + scalp + MRI +
source data, make MNE Scan production-ready, land the cortical workflow in
mne_analyze, and close the remaining MNE-Python algorithm parity items.**

### Pillars

| Pillar | Tasks | Outcome |
|--------|-------|---------|
| **★ 0. MNE Inspect multimodal visualization** | TASK 10 | sEEG, ECoG, scalp EEG, MEG, MRI slices and source overlays render together in MNE Inspect with picking, data overlays, and BIDS loading |

## TASK 1 — mne-mna Python Package for MNA/MNX Interoperability

**Goal:** Create a Python package (`mne-mna`) that reads and writes `.mna`
(JSON) and `.mnx` (CBOR) files, integrating with MNE-Python workflows. This
enables Python-based researchers to consume and produce MNA project files,
bridging the MNE-CPP and MNE-Python ecosystems.

### Rationale for v2.3.0 (not v2.2.0)

- The MNA/MNX format should be published and citable (v2.2.0 TASK 17) before
  approaching the MNE-Python community for adoption.
- Coordination with MNE-Python maintainers is needed for API design review
  and potential joint publication.
- One more release cycle allows the format to stabilize based on internal
  usage and feedback.

### Package scope

#### Core I/O

| Function | Description |
|----------|-------------|
| `mne_mna.read_project(path)` | Read `.mna` (JSON) or `.mnx` (CBOR) → `MnaProject` object |
| `mne_mna.write_project(project, path)` | Write `MnaProject` → `.mna` or `.mnx` (auto-detected by extension) |
| `mne_mna.mna_to_mnx(in_path, out_path)` | Convert JSON ↔ CBOR |

#### Data model (Python classes mirroring C++)

| Class | Maps to C++ |
|-------|-------------|
| `MnaProject` | `MNALIB::MnaProject` |
| `MnaGraph` | `MNALIB::MnaGraph` |
| `MnaNode` | `MNALIB::MnaNode` |
| `MnaPort` | `MNALIB::MnaPort` |
| `MnaSubject` | `MNALIB::MnaSubject` |
| `MnaSession` | `MNALIB::MnaSession` |
| `MnaRecording` | `MNALIB::MnaRecording` |
| `MnaFileRef` | `MNALIB::MnaFileRef` |
| `MnaScript` | `MNALIB::MnaScript` |
| `MnaVerification` | `MNALIB::MnaVerification` |
| `MnaParamTree` | `MNALIB::MnaParamTree` |
| `MnaParamBinding` | `MNALIB::MnaParamBinding` |
| `MnaOpSchema` | `MNALIB::MnaOpSchema` |

Enums: `MnaDataKind`, `MnaFileRole`, `MnaNodeExecMode`, `MnaPortDir` as
Python `enum.Enum` classes with identical string mappings.

#### MNE-Python integration

| Feature | Description |
|---------|-------------|
| `project.to_raw(node_id)` | Extract `mne.io.Raw` from a project's file refs |
| `project.to_epochs(node_id)` | Extract `mne.Epochs` |
| `project.to_source_estimate(node_id)` | Extract `mne.SourceEstimate` |
| `project.add_raw(raw, subject, session, recording)` | Add an `mne.io.Raw` as a file ref |
| `MnaGraph.to_networkx()` | Export DAG as a NetworkX graph for visualization |
| `MnaGraph.from_networkx(G)` | Import DAG from NetworkX |

#### BIDS integration

| Feature | Description |
|---------|-------------|
| `mne_mna.from_bids(bids_path)` | Create `MnaProject` from an existing BIDS dataset (using `mne_bids`) |
| `mne_mna.to_bids(project, bids_root)` | Export file refs into BIDS directory structure |

### Dependencies

- `cbor2` — CBOR encoding/decoding (for `.mnx`)
- `mne` — MNE-Python (optional; required for `to_raw()` etc.)
- `mne_bids` — MNE-BIDS (optional; required for BIDS integration)

### Package structure

```
mne-mna/
  pyproject.toml
  README.md
  LICENSE
  src/
    mne_mna/
      __init__.py
      _project.py      # MnaProject, MnaSubject, MnaSession, etc.
      _graph.py         # MnaGraph, MnaNode, MnaPort
      _io.py            # read_project, write_project, mna_to_mnx
      _types.py         # Enums (MnaDataKind, MnaFileRole, etc.)
      _script.py        # MnaScript
      _verification.py  # MnaVerification, MnaProvenance
      _params.py        # MnaParamTree, MnaParamBinding
      _schema.py        # MnaOpSchema
      _mne_bridge.py    # to_raw, to_epochs, etc. (optional mne dep)
      _bids_bridge.py   # from_bids, to_bids (optional mne_bids dep)
  tests/
    test_io.py
    test_roundtrip.py
    test_mne_bridge.py
    test_bids_bridge.py
    conftest.py
```

### Cross-validation with MNE-CPP

- Round-trip test: C++ writes `.mna` → Python reads → Python writes → C++
  reads → verify identical.
- Same for `.mnx` (CBOR).
- Use `resources/examples/` MNA files as shared test fixtures.

### Publication coordination

- Contact MNE-Python core team (Gramfort, Larson, Engemann) after v2.2.0
  release with the published format paper DOI.
- Propose `mne-mna` as a companion package (similar to `mne-bids`,
  `mne-connectivity`, `mne-icalabel`).
- Joint publication opportunity: MNA/MNX as a reproducible analysis container
  format for the MNE ecosystem.

### Acceptance criteria

1. `pip install mne-mna` installs cleanly on Python 3.10+.
2. Round-trip `.mna` read/write preserves all fields.
3. Round-trip `.mnx` read/write preserves all fields including embedded binary data.
4. `MnaProject` ↔ MNE-Python object conversion works for Raw, Epochs, and
   SourceEstimate.
5. `from_bids()` correctly imports a BIDS dataset into MNA project structure.
6. All enum string mappings match C++ implementation exactly.
7. CI cross-validates Python ↔ C++ round-trips on every commit.

## TASK 2 — NeuralSet Interoperability & Unified Event Model

**Carried over from v2.2.0 TASK 33** (deferred 26 April 2026). Meta released
NeuralSet in April 2026; full interop is a substantial late-binding feature
that should not block v2.2.0.

### Scope summary

- **`MnaEventTable`** — flat event table mirroring NeuralSet's Event model
  (`type`, `start`, `duration`, `timeline`, `extra`); JSON + CBOR I/O; CSV /
  Parquet export; embedded in `MnaRecording`.
- **Extended `MnaDataKind` / `MnaFileRole`** — add `Fmri`, `Fnirs`, `Spikes`,
  `Audio`, `Video`, `Image`, `Embedding`, `EventTable`.
- **Op registry tags** — add `tags` array to `MnaOpSchema`; categorise built-in
  ops as `loader` / `transform` / `extractor` / `external`.
- **`segment_events` op** — declarative event-locked segmenter (NeuralSet
  `Segmenter` parity).
- **`MnaExporter`** — BIDS-aligned event CSV export, BIDS layout export,
  NeuralSet `Chain.from_json()` export.
- **`MnaImporter`** — BIDS dataset → `MnaProject` and `*_events.tsv` →
  `MnaEventTable` importers.
- **BIDS-aligned timelines** — add `task` and `run` fields to `MnaRecording`;
  timeline string `"sub-<id>_ses-<id>_task-<task>_run-<run>"`.

### Reference

Full design specification lives in
[v2.2.0-requirements.md § TASK 33](v2.2.0-requirements.md#task-33--neuralset-interoperability--unified-event-model--deferred-to-v230)
— the design carries over verbatim as the v2.3.0 baseline.

### Dependencies

- v2.2.0 must be released first (NeuralSet interop should target a stable
  MNA/MNX format).
- Coordination with NeuralSet maintainers (Meta / FAIR — Jean-Rémi King team).

### Acceptance criteria

1. `MnaProject` round-trips through NeuralSet's BIDS layout without data loss.
2. NeuralSet `Study.load()` consumes `MnaExporter::exportBidsLayout()` output.
3. `MnaEventTable::writeCsv()` output is consumed by NeuralSet's events frame.
4. `segment_events` op produces `Epochs` matching MNE-Python `Epochs(...)` for
   the same trigger/window definition.
5. CI cross-validates a NeuralSet → MNA → NeuralSet round-trip.

## TASK 3 — MNE Scan Production Hardening (Pillar A)

**Goal:** Promote `mne_scan` from "real-time research platform" to a
turn-key production application that a clinician or lab tech can install,
configure, and run without source-code knowledge. v2.2.0 already wired
MNX-as-project-storage and modernised the pipeline canvas — v2.3.0 closes
the remaining usability/coverage gaps so the binary ships in the v2.3.0
release announcement as a headline application.

### Sub-task 3.1 — Plugin coverage audit & gap fill

Inventory existing acquisition plugins and document supported hardware in a
single matrix (`doc/website/docs/manual/mne-scan-hardware.mdx`). Confirm
each plugin builds and loads on macOS / Linux / Windows; mark any that are
broken or stale. Existing plugins:

`babymeg`, `brainamp`, `eegosports`, `fiffsimulator`, `ftbuffer`, `gusbamp`,
`hpi`, `lsladapter`, `natus`, `tmsi` (sensor) and `averaging`, `covariance`,
`neuronalconnectivity`, `noisereduction`, `rtcmne`, `rtfwd`, `writetofile`
(processing).

Acceptance: hardware matrix page committed; all plugins build green on all
3 desktop platforms; broken plugins either fixed or explicitly marked
`UNMAINTAINED` in `CMakeLists.txt`.

### Sub-task 3.2 — RT-CMNE plugin wiring

`InvCMNE` shipped in v2.2.0 with the `mne_compute_cmne` CLI tool. The
`rtcmne` plugin currently runs the classical real-time MNE; this task
adds a "CMNE" mode toggle in its settings widget, loading a trained
`.onnx` model from a file picker and switching the per-block inverse to
`InvCMNE::compute()`.

Acceptance: `rtcmne` plugin exposes `Method = MNE | dSPM | sLORETA | CMNE`;
when CMNE is selected the model path is required and validated;
`tests/applications/test_mne_scan/test_rtcmne_cmne.cpp` runs the plugin
with a fixture model and verifies output dimensions match.

### Sub-task 3.3 — Recording, replay, and project resume

- `writetofile` plugin: confirm round-trip with `fiffsimulator` and add a
  "Recording…" status indicator to the main toolbar (file size, elapsed
  time).
- `fiffsimulator`: support replaying any FIFF previously written by
  `writetofile`, including SSP and bad-channel state.
- Project resume: when a `.mna`/`.mnx` project is opened, restore not only
  the pipeline but also the **dock-widget layout, last selected channel
  view, and last filter settings** of every plugin (via `saveSettings()`/
  `loadSettings()` already on `IPlugin`).

Acceptance: round-trip `record → close → open project → replay` works on
all 3 desktop platforms; layout persistence verified by manual smoke test
checklist in `doc/website/docs/manual/mne-scan-quickstart.mdx`.

### Sub-task 3.4 — Quickstart + clinical-deployment documentation

New manual pages under `doc/website/docs/manual/`:

- `mne-scan-quickstart.mdx` — install, choose hardware, build a 3-node
  pipeline (sensor → filter → averaging), record, replay.
- `mne-scan-hardware.mdx` — supported devices, driver setup per OS.
- `mne-scan-deployment.mdx` — packaging, autostart, service mode.

Acceptance: pages link from the website sidebar; quickstart is verified
end-to-end against the installed binary on at least one platform per
maintainer.

### Sub-task 3.5 — Installer / packaging polish

Ensure `mne_scan` ships in the `v2.3.0` GitHub Release artifacts on all 3
desktop platforms with all hardware plugin DLLs/dylibs/sos, the bundled
`default.mna` template (sensor → filter → display), and a code-signed
macOS `.dmg` if signing certs are available.

Acceptance: download → install → launch → run `default.mna` works without
extra installation steps; release notes screenshot the running pipeline.

### Acceptance criteria (TASK 3 overall)

1. Hardware matrix page documents every shipped plugin's status per OS.
2. RT-CMNE works end-to-end with a real-time MEG/EEG simulated stream.
3. Project save/restore preserves pipeline, plugin settings, and dock
   layout.
4. Quickstart manual takes a new user from zero → recording in < 15 min.
5. v2.3.0 release artefacts include `mne_scan` with bundled plugins on
   macOS, Linux, Windows.

## TASK 4 — mne_analyze Cortical Analysis Workflow (Pillar B)

**Goal:** Land the highest-priority `mne_analyze` parity items from
[gap-analysis.md § 14](gap-analysis.md#14-gui-application-mne_analyze-vs-mne-c-svn-mne_analyze)
so the GUI supports the canonical *load surface → load STC → pick vertex →
read timecourse* clinical workflow that is the most-used path in the
legacy MNE-C `mne_analyze`.

This task is the user-visible counterpart to v2.2.0's library work
(MxNE, Gamma-MAP, CMNE, source-space cluster permutation tests): the
algorithms exist; the GUI now needs to expose them.

### Sub-task 4.1 — FreeSurfer cortical surface plugin

New plugin `src/applications/mne_analyze/plugins/cortical_surface/`:

- "Load Surface…" menu action with `SUBJECTS_DIR` browser; loads
  `lh.inflated`, `rh.inflated`, `lh.pial`, `rh.pial`, `lh.white`,
  `rh.white` via existing `FsSurface`.
- 3D rendering re-uses `BrainSurface` QRhi pipeline.
- Hemisphere toggle (`L | R | Both`) and surface-type selector
  (`inflated | pial | white`) in the controls dock.
- Curvature shading (sulci/gyri) using `FsSurface::curvature()`.

### Sub-task 4.2 — STC overlay loading & rendering

- "Load Source Estimate…" menu action; reads `.stc` (already in `MNE`
  library) into a new `SourceEstimateModel`.
- Heat/cool colour maps with `fthresh / fmid / fmax` controls
  (mirroring MNE-C `mne_analyze` overlay controls).
- Time slider drives current sample; "Play" button animates through time
  points.
- Smoothing-step control (uses `MNECorticalMap::smoothOperator()` from
  v2.2.0).

### Sub-task 4.3 — Vertex picking with timecourse readout

- 3D pick on the cortical surface returns the nearest vertex.
- New "Source Time Course" dock displays the full STC time series at the
  picked vertex with a vertical cursor synced to the time slider.
- Multiple picks accumulate in a "Time Course Manager" list (name,
  colour, hemisphere, vertex index, save/export to CSV).

### Sub-task 4.4 — Source estimation in the GUI

New "Compute Source Estimate" dialog driven by an existing inverse
operator (or computed on the fly from a forward + cov + evoked already
loaded in the BIDS tree):

- Method dropdown: `MNE | dSPM | sLORETA | eLORETA | MxNE | Gamma-MAP |
  CMNE`.
- SNR / λ² control for linear methods; sparsity hyper-parameter for
  sparse methods; ONNX model path for CMNE.
- Output STC populates the `SourceEstimateModel` so it overlays on the
  loaded surface immediately.

### Sub-task 4.5 — Label management

New "Labels" dock:

- Load a single `.label` or batch-load `.annot` (Desikan-Killiany,
  Destrieux); reuses `FsAtlasLookup` from v2.2.0.
- Toggle per-label visibility on the surface.
- Right-click label → "Extract Time Course" with collapse method
  (`mean | max | L2 | L1 | mean+sign-flip`); appends to the Time Course
  Manager.
- "Create Label from Marked Vertices…" — interactive vertex marking +
  flood-fill, save as `.label`.

### Sub-task 4.6 — Field mapping (stretch goal)

If time permits in the cycle:

- MEG field interpolation on helmet via spherical-harmonic basis
  (re-uses SSS code in `MNE/sss`).
- EEG potential map on scalp surface using existing `FieldMap` class.

### Acceptance criteria (TASK 4 overall)

1. Surface plugin loads `lh.inflated`/`rh.inflated` from a sample subject
   and renders both hemispheres in < 2 s.
2. STC overlay loads, animates through time, and respects
   `fthresh/fmid/fmax` colour scaling.
3. Vertex picking returns the correct vertex and displays the matching
   timecourse from the loaded STC.
4. Source estimation dialog produces an STC equivalent to the
   `mne_compute_mne` CLI tool for the same inputs.
5. Label loader accepts both `.label` and `.annot`; extracted timecourses
   match `mne_extract_label_timecourse` CLI output.
6. Workflow documented in `doc/website/docs/manual/mne-analyze-cortical.mdx`.

## TASK 6 — MNE-Python High-Priority Algorithm Parity (Pillar D)

**Goal:** Close the three highest-priority MNE-Python algorithm gaps
identified in [gap-analysis.md](gap-analysis.md):

| Algorithm | Gap-analysis section | Target library | Notes |
|-----------|----------------------|----------------|-------|
| **TF-MxNE** | § 1 | `inv/sparse_source_estimators` | Time-frequency mixed-norm; extends v2.2.0 MxNE with TF dictionary |
| **Standard montages (10-20, 10-10, 10-05)** | § 4 | `fiff` (new `FiffStandardMontage`) | Built-in electrode position databases — unblocks EEG users without dig points |
| **Maxwell movement compensation** | § 6 | `dsp` (extend `MaxwellFilter`) | Head-position-aware tSSS — uses CHPI traces from `RtHpi` |

### Sub-task 6.1 — TF-MxNE

- New `TFMxNESolver` in `src/libraries/inv/sparse_source_estimators/`.
- Time-frequency dictionary via Gabor atoms (re-use `dsp/morlet_wavelet`).
- L1+L2 mixed-norm penalty, FISTA optimisation.
- CLI tool `mne_compute_tfmxne` mirroring `mne_compute_cmne` pattern.
- Cross-validation against `mne.inverse_sparse.tf_mixed_norm()`.

### Sub-task 6.2 — Standard montages

- New `FiffStandardMontage` class loading Cartesian electrode coordinates
  for 10-20, 10-10, 10-05, biosemi-{16,32,64,128}, easycap-{M1,M10}.
- YAML/JSON resource files under `resources/montages/`.
- `Raw::setMontage(const FiffStandardMontage&)` populates `dig_points`
  and channel locations.
- CLI tool `mne_apply_montage` for batch application.
- Cross-validation against `mne.channels.make_standard_montage()`.

### Sub-task 6.3 — Maxwell movement compensation

- Extend `MaxwellFilter` to accept a per-block head-position trajectory
  (output of `RtHpi`).
- Recompute the SSS basis at each head position; interpolate between
  basis updates.
- Cross-validation against `mne.preprocessing.maxwell_filter(..., head_pos=...)`
  on a sample CHPI dataset.

### Acceptance criteria

1. TF-MxNE STC output matches MNE-Python within 1e-4 RMSE on the sample
   dataset.
2. `mne_apply_montage` produces identical channel locations to
   `make_standard_montage()` for all shipped montages.
3. Maxwell movement-compensated raw matches MNE-Python output within
   1e-3 relative error.
4. Each new algorithm has a unit test, CLI test, and Python
   cross-validation test (`MNE_REQUIRE_PYTHON=true`).

## TASK 9 — MNE-Python Simulation API (Pillar D)

**Goal:** Port the MNE-Python simulation API from
[gap-analysis.md § 11](gap-analysis.md#11-simulation) — currently mne-cpp
only has the `mne_simu` CLI tool and the `fiffsimulator` plugin.

### Scope

- New `src/libraries/sim/` library with:
  - `simulateStc()` / `simulateSparseStc()` — generate brain activity on
    a source space.
  - `simulateRaw()` — synthetic continuous data from a forward solution.
  - `simulateEvoked()` — synthetic averaged responses with noise.
  - `addNoise()`, `addEcg()`, `addEog()`, `addChpi()` — artifact injection.
- CLI tools `mne_simulate_raw`, `mne_simulate_evoked`.
- Cross-validation against `mne.simulation.*`.

### Acceptance criteria

1. `simulateRaw()` output matches `mne.simulation.simulate_raw()` within
   1e-5 RMSE for the same forward + STC + noise covariance.
2. `simulateEvoked()` matches `mne.simulation.simulate_evoked()`.
3. CLI tools shipped in `src/tools/simulation/`.

## ★ TASK 10 — MNE Inspect Multimodal Visualization (Pillar 0 — HIGHEST PRIORITY)

**Goal:** Make MNE Inspect the **multimodal viewer for intracranial + scalp +
MRI + source data**. v2.2.0 shipped the disp3D primitives (`ElectrodeObject`
for sEEG depth shafts and ECoG grids/strips, `SliceObject` for MRI slices,
plus the existing `BrainSurface`, `DipoleObject`, `NetworkGraph`, sensor and
helmet renderables) but **none of them are integrated into MNE Inspect**.
This task wires every modality into a single coherent 3-D scene with picking,
data-driven colouring, BIDS loading, and a debug pass on each renderable.

This is the **single most important user-facing feature of v2.3.0** — it
turns mne-cpp from "we have the libraries" into "intracranial researchers
have a working desktop app".

### Scope summary

| Modality | Renderable (already in disp3D) | New in MNE Inspect |
|----------|-------------------------------|--------------------|
| **sEEG** depth electrodes | `ElectrodeObject` (shaft + contacts, instanced) | Loader, picking, contact data overlay, label panel |
| **ECoG** grids / strips | `ElectrodeObject` (extend for grid topology) | Loader, picking, grid-aware data overlay |
| **Scalp EEG** sensors | `SensorPositionsObject` / `SensorSurfaceObject` | Loader from FIFF info / BIDS |
| **MEG** sensors + helmet | existing helmet renderable | Toggle from FIFF info |
| **MRI** orthogonal slices | `SliceObject` | Loader, axial/coronal/sagittal sliders, intensity windowing |
| **Cortical surface** | `BrainSurface` | Loader (FreeSurfer subject browser) |
| **Source overlay** | existing surface activation overlay | STC loader (shared with TASK 4) |
| **Dipoles** | `DipoleObject` | Loader from `.dip` |
| **Connectivity** | `NetworkGraph` | Loader from existing connectivity output |

### Sub-task 10.1 — `Electrodes` plugin (sEEG + ECoG)

New plugin `src/applications/mne_inspect/app/plugins/electrodes/`:

- "Load Electrodes…" menu action accepting:
  - BIDS `*_electrodes.tsv` (uses `BIDSLIB::BidsElectrode::readTsv()`).
  - Plain CSV with columns `name,x,y,z[,type,size,group]`.
  - FIFF dig points where channel kind is `FIFFV_SEEG_CH` or `FIFFV_ECOG_CH`.
- Auto-detect modality:
  - `type == "depth"` → render as sEEG shaft (existing path).
  - `type == "grid" | "strip"` → render as ECoG: spheres on a flexible 2-D
    quad mesh; reuse `ElectrodeObject` with `shaftRadius = 0.0` and add an
    optional triangle strip linking the contact instances ("grid mode").
- Per-shaft / per-grid colour and visibility toggles in a tree dock.
- Selection in 3-D ↔ selection in the channel list (bidirectional).

#### Sub-task 10.1a — Extend `ElectrodeObject` for ECoG topology

The current `ElectrodeObject` assumes a linear shaft. Extend with:

```cpp
enum class ElectrodeLayout { Depth, Strip, Grid };

struct ElectrodeArray {
    QString          label;
    ElectrodeLayout  layout = ElectrodeLayout::Depth;
    int              gridRows = 1;   // for Grid; 1 for Strip / Depth
    int              gridCols = 1;
    QVector<ElectrodeContact> contacts;
    float            shaftRadius = 0.4f;   // Depth only
    QColor           shaftColor;
};
```

Rename `ElectrodeShaft` → `ElectrodeArray` (deprecation alias kept), update
the renderer:
- `Depth` → existing cylinder + sphere instances.
- `Strip` → sphere instances only, no cylinder.
- `Grid` → sphere instances + optional translucent quad mesh through the
  contact positions (Catmull-Clark or simple bilinear surface) for a visual
  reference of the grid.

### Sub-task 10.2 — `MriSlices` plugin (MRI ortho viewer in 3-D)

New plugin `src/applications/mne_inspect/app/plugins/mri_slices/`:

- "Load MRI…" accepts `.mgh`, `.mgz`, `.nii`, `.nii.gz` via `MRILIB::MriVolData`.
- Three `SliceObject` instances (axial, coronal, sagittal) with sliders in
  the controls dock; voxel-to-world transform applied so slices align with
  surfaces and electrodes.
- Intensity window/level sliders; greyscale or hot/cool colourmap.
- "Show Picked Vertex in MRI Slices" — when a vertex is picked on a
  cortical surface or a contact is picked on an electrode, all three slices
  jump to that voxel.

### Sub-task 10.3 — Scene composition + picking layer

Audit `mne_inspect/app/View3D` (or whichever 3-D scene wrapper exists) and:

- Implement object-id picking (already present for some renderables) for

### Sub-task 10.4 — Data-driven colouring across modalities

- Source estimate value at vertex.
- MRI intensity at voxel.
- Time slider scrubs all overlays simultaneously (shared timeline).
- Coordinated colourmap selector with `fmin/fmid/fmax` / sequential / divergent
  modes.

### Acceptance criteria (TASK 10 overall)

1. Load a BIDS sample with sEEG + ECoG + MRI + cortical surface; all four
   modalities render in one 3-D scene with correct relative geometry.
2. Click any sEEG or ECoG contact → contact name, position, and current
   data value appear in the Pick dock; the contact is highlighted.
3. Click a cortical vertex → vertex index, position, and current source
   estimate value appear; MRI slices jump to that voxel.
4. Toggle each modality on/off independently; no QRhi warnings or leaks.
5. Apply iEEG amplitude overlay → contact colours match the colour-bar
   for known min/max values within 1 % tolerance.
6. Render performance: ≥ 30 FPS for `cortical surface (327k tris) + 120
   sEEG contacts + 64-contact ECoG grid + 3 MRI slices` on a reference
   2024-class laptop.
7. WASM build of `mne_inspect` opens the same sample (with appropriate
   browser file picker) and renders all modalities.
8. Screenshot regression test passes on macOS / Linux / Windows.
9. Multimodal manual page committed and linked from website sidebar.
