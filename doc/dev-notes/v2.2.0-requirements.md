## Conventions & Patterns for All New Code

Before implementing any task, follow these project-wide rules:

### File layout
- License header: BSD-3-Clause, copyright "Christoph Dinh" (or original authors
  if porting). Use the `@file`, `@author`, `@since 2.2.0`, `@date` Doxygen block
  found in every existing header (see `src/libraries/dsp/welch_psd.h` for the
  current template).
- Include guard: `#ifndef FILENAME_H` / `#define FILENAME_H`.
- Section separators: the `//====...` bar (96 `=` chars) between logical blocks.

### Naming
- **Library directory**: lowercase, ≤5 letters (e.g. `mna`, `ml`, `sts`).
- **CMake target**: `mne_<dir>` (e.g. `mne_mna`, `mne_ml`, `mne_sts`).
- **Namespace**: `<DIR>LIB` all-caps (e.g. `MNALIB`, `MLLIB`, `STSLIB`).
- **Export macro**: `<DIR>SHARED_EXPORT` (e.g. `MNASHARED_EXPORT`, `STSSHARED_EXPORT`).
- **Global header**: `<dir>_global.h` / `<dir>_global.cpp`.
- **Class prefix**: capitalised `<Dir>` (e.g. `MnaProject`, `MlModel`,
  `StsCluster`). Exception: FIFF classes use `Fiff` prefix.
  Stats classes keep the `Stats` prefix for readability (e.g. `StatsTtest`,
  `StatsCluster`) even though the library directory is `sts`.
- **Filenames**: all lowercase with underscores matching the class
  (e.g. `MnaProject` → `mna_project.h` / `mna_project.cpp`).

### CMakeLists.txt for a new library
Follow the pattern in `src/libraries/dsp/CMakeLists.txt`:
1. `cmake_minimum_required(VERSION 3.14)` + `project(mne_<dir> LANGUAGES CXX)`.
2. Enable `CMAKE_AUTOUIC`, `CMAKE_AUTOMOC`, `CMAKE_AUTORCC`.
3. Find Qt components (`Core` minimum; add others only if needed).
4. `set(SOURCES ...)` / `set(HEADERS ...)` — list every `.cpp` / `.h`.
5. `set(FILE_TO_UPDATE <dir>_global.cpp)` + the `OBJECT_DEPENDS` trick.
6. `add_library(${PROJECT_NAME} ${SOURCES} ${HEADERS})`.
7. `target_include_directories(... PUBLIC $<BUILD_INTERFACE:...>)`.
8. `target_link_libraries(... PRIVATE Qt6::Core eigen <dependency targets>)`.
9. `target_compile_definitions(... PRIVATE MNE_<DIR>_LIBRARY ...)`.
10. `install(TARGETS ...)` and `install(FILES ${HEADERS} ...)`.
After creating the library, add `add_subdirectory(<dir>)` in
`src/libraries/CMakeLists.txt` at the correct dependency level.

### Training / Python bridge
All ML model **training** is done in Python (PyTorch, scikit-learn, etc.) and
launched from C++ via the `PythonRunner` utility in `src/libraries/utils/`.
The `MLTrainer` convenience class in the `ml` library wraps `PythonRunner`
with ML-specific helpers (prerequisite checking, venv management, progress
streaming). No training methods exist in the C++ model classes themselves —
only `predict()`, `save()`, and `load()`.

### Test conventions
Unit tests live in `src/testframes/` mirroring the library path.
Each test links `mne_<dir>` + `Qt6::Test` and uses `QTest` macros.

**Coverage target: 75% line coverage across all new and modified libraries.**

All tests must be **meaningful and data-driven**:
- Use real MNE-CPP test data from `mne-cpp-test-data/` (FIFF files, surfaces,
  source spaces, forward solutions, covariance matrices) — not just trivial
  synthetic matrices.
- When real data is impractical (e.g. permutation tests), use carefully
  constructed synthetic data with known ground truth and verify against
  published reference values or cross-validated with MNE-Python/SciPy outputs.
- Every public method must have at least one test that exercises it with
  realistic inputs.
- Edge cases: empty inputs, single-channel data, rank-deficient covariance,
  zero-length epochs.
- Performance assertions where applicable (e.g. "PSD of 64ch × 60s at 1kHz
  completes in < 2s").
- Each test file should include a brief comment at the top explaining what
  test data it uses and what ground truth it validates against.

**Never skip tests — fix instead.** If a test fails or data is missing, do not
use `QSKIP()` to paper over it. Diagnose the root cause and fix the test or the
code under test. A skipped test is an invisible hole in coverage. The only
acceptable use of `QSKIP()` is when the entire test-data directory is absent
(CI matrix job that intentionally runs without data).

**Ground implementations on MNE-SVN (C) and MNE-Python wherever possible.**
Both codebases live in this workspace and serve as reference implementations:
- **MNE-SVN** (`/MNE/MNE/SVN-MNE/`): Original C library by Hämäläinen.
  Key files: `MNE-libs/mne/mne_inverse_io.c`, `mne_inverse_util.c`,
  `mne_cov_matrix.c`, `mne_whiten.c`, `mne_source_space.c`,
  `mne_forward_util.c`, `mne_stc_util.c`. Tools: `mne_inverse_operator`,
  `mne_compute_mne`, `mne_compute_raw_inverse`, `mne_make_source_space`.
- **MNE-Python** (`/mne-python/mne/`): Python ecosystem.
  Key module: `mne.minimum_norm` (`apply_inverse`, `make_inverse_operator`,
  `read_inverse_operator`). Tests: `mne/minimum_norm/tests/test_inverse.py`
  contains numerical ground-truth checks:
  - dSPM output range `7.5 < max < 15`;
  - sLORETA output range `2 < max < 7`;
  - Localization bias tests with known dipole (dSPM fixed-ori → 96–98%
    correct, sLORETA → 100%).
  Test data: `sample_audvis_trunc-meg-eeg-oct-4-fwd.fif`,
  `sample_audvis_trunc-meg-eeg-oct-4-meg-inv.fif`, `sample_audvis_trunc-ave.fif`,
  `sample_audvis_trunc-cov.fif`, `sample_audvis_trunc_raw.fif`. SNR=3, λ²=1/9.

When implementing an algorithm that exists in MNE-Python or MNE-SVN C, the test
should validate that the mne-cpp output matches the reference output within
stated tolerances (e.g. `assert_allclose(rtol=1e-5)` equivalent). If exact
matching is impossible (e.g. different floating-point paths), document the
expected deviation and set a sensible tolerance.

**Live cross-validation via PythonRunner (TASK 16):**
Instead of pre-computing MNE-Python reference values and hard-coding them as
test constants, test frames should use `PythonRunner` to run MNE-Python scripts
**on-the-fly** and compare the results directly. This keeps cross-validation
always up-to-date with the current MNE-Python version and avoids stale embedded
values. The pattern is:

```cpp
// In a C++ test frame:
PythonRunner py;
auto result = py.runCode(R"(
    import mne, numpy as np, json, sys
    fwd = mne.read_forward_solution('path/to/fwd.fif')
    cov = mne.read_cov('path/to/cov.fif')
    evoked = mne.read_evokeds('path/to/ave.fif')[0]
    inv = mne.minimum_norm.make_inverse_operator(evoked.info, fwd, cov)
    stc = mne.minimum_norm.apply_inverse(evoked, inv, lambda2=1./9., method='dSPM')
    # Output as JSON for C++ to parse
    json.dump({'data_max': float(stc.data.max()),
               'data_shape': list(stc.data.shape),
               'peak_vertex': int(np.argmax(stc.data.max(axis=1)))}, sys.stdout)
)");
// Parse result.stdOut as JSON, compare with MNE-CPP output
```

When Python is unavailable (CI job without MNE-Python), the test falls back to
embedded reference values but logs a warning. This ensures CI never breaks even
when MNE-Python is not installed, while giving developers the full live
cross-validation when running locally. A `PythonTestHelper` utility class
(TASK 16) provides convenience methods for this pattern.

**Coverage measurement**: CI runs `gcov`/`lcov` with `WITH_CODE_COV=ON`.
The 75% target applies per-library; new libraries (`mna`, `ml`, `sts`)
must ship with tests that meet this bar before merge.

## TASK 1 — New Library: `mna` (Analysis Container Format)

### Goal
Create the `mna` library (`src/libraries/mna/`) that reads and writes `.mna`
(JSON) and `.mnx` (CBOR) project container files. A container bundles references
to FIFF raw data, forward solutions, inverse operators, covariance matrices,
source estimates, BEM surfaces, and FreeSurfer reconstructions into a single
project manifest with full provenance tracking, transparent parameter storage,
and — critically — an **executable computational graph** inspired by the ONNX
graph model. The graph stitches together files, processing operations, and
intermediate results into a directed acyclic graph (DAG) that can be validated,
serialized, and executed as a complete analysis pipeline.

Beyond offline batch processing, the graph supports **real-time data streams**
as first-class port types, making `.mna` files usable as **configuration files
for MNE Scan** (the real-time acquisition/processing application). Nodes can
also delegate execution to **external processes via IPC** (FreeSurfer, MRI
acquisition, custom tools), and a **parameter tree** with formula-driven bindings
allows attributes to be dynamically adjusted at runtime based on upstream outputs
or streaming metrics.

### Why — The Decluttering Problem
A single source estimation analysis today requires the user to manage a staggering
number of separate files scattered across potentially different directories on
different machines:

- **MRI data**: T1 volume (`.mgz`), FreeSurfer surfaces (pial, white, inflated),
  annotations, labels, BEM surfaces, volume parcellations
- **Raw M/EEG**: one or more `.fif` files (often gigabytes each)
- **Events/annotations**: event files, annotation sidecars
- **Preprocessing outputs**: SSP projectors, ICA solutions, bad channel lists
- **Covariance matrices**: noise covariance, data covariance
- **Forward model**: source space, BEM solution, forward operator
- **Inverse operator**: inverse solution, regularisation parameters
- **Source estimates**: STC files, one per condition/epoch
- **Connectivity**: network results

On top of scattered files, each processing step involves **parameters** (e.g.
regularisation λ=1/9, noise covariance method=Ledoit-Wolf, source space=oct-6,
BEM layers=3) that are currently invisible — buried in command-line flags or
forgotten entirely. Reproducing someone else's analysis (or your own from 6 months
ago) requires detective work.

The `.mna`/`.mnx` format solves both problems:
1. **Single manifest**: one file links every artifact in the pipeline together
   with relative paths, regardless of where the actual files live on disk.
2. **Transparent parameters**: every processing step records its tool, version,
   and full parameter dictionary so the pipeline is self-documenting.
3. **Packaging (v2.2.0+)**: the `.mnx` binary format is designed to evolve into
   a true container archive (like MP4 for video) that embeds all referenced
   files directly, eliminating the need for external file links entirely.

### Computational Graph Model (ONNX-inspired)

Beyond file management and provenance, the `.mna`/`.mnx` format implements an
**executable computational graph** modeled after the ONNX graph specification.
Just as ONNX stitches tensor operations (Conv, Gemm, Relu, …) into a DAG where
each node consumes named inputs and produces named outputs, the MNA graph
stitches neuroimaging operations (filter, compute_forward, apply_inverse, …)
into a DAG where each node consumes and produces typed data references.

**Key parallels with ONNX:**

| ONNX concept | MNA equivalent | Purpose |
|--------------|----------------|---------|
| `Graph` | `MnaGraph` | Top-level DAG container with named graph inputs/outputs |
| `Node` | `MnaNode` (replaces `MnaStep`) | One operation: op type + input ports + output ports + attributes |
| `Node.op_type` | `MnaNode::opType` | String naming the operation (looked up in the op registry) |
| `Node.input[]` / `Node.output[]` | `MnaPort` references | Typed, named input/output slots connected by edges |
| `AttributeProto` | `MnaNode::attributes` (`QVariantMap`) | Operation parameters (λ², method, spacing, …) |
| `ValueInfoProto` | `MnaPort` | Typed port descriptor: name + `MnaDataKind` + optional shape |
| `OpSchema` | `MnaOpSchema` | Contract declaring an op's expected inputs, outputs, and attributes |
| `OpSchemaRegistry` | `MnaOpRegistry` | Central catalog of all known operation schemas |
| `GraphProto.initializer` | Graph-level `MnaFileRef` inputs | Constant data fed into the graph (raw files, MRI, etc.) |
| *(no equivalent)* | `MnaParamTree` + `MnaParamBinding` | Dynamic parameter bindings with formula evaluation and trigger conditions |
| *(no equivalent)* | `MnaNode::execMode = Stream` | Real-time streaming execution (graph as MNE Scan config) |
| *(no equivalent)* | `MnaNode::execMode = Ipc` | Delegation to external processes (FreeSurfer, scanners, etc.) |

**What the graph model enables:**

1. **Composable pipelines**: Build an analysis by connecting nodes. The output
   port of a "compute_covariance" node feeds the input port of an
   "make_inverse_operator" node. Parameters are explicit on each node.
2. **Validation before execution**: The graph executor can verify, before
   running anything, that all input ports are connected, data types match,
   required attributes are present, and the graph is acyclic.
3. **Selective re-execution**: When a parameter changes on one node, only
   that node and its downstream dependents need re-running — the graph
   knows the dependency structure.
4. **Serialization**: The entire graph (topology + parameters + intermediate
   result references) round-trips through `.mna` (JSON) and `.mnx` (CBOR),
   making analyses fully reproducible and shareable.
5. **Intermediate results**: Each node's output ports can reference cached
   results (files on disk or embedded in `.mnx`), enabling incremental
   execution and result inspection at any pipeline stage.
6. **Real-time streaming**: Ports with `RealTimeStream` data kind connect to
   live data channels (MEG acquisition, LSL, FIFF real-time server). This
   turns an `.mna` file into a **configuration file for MNE Scan** — declaring
   acquisition sources, processing chains, and output routing in one document.
7. **Heterogeneous execution**: Nodes can delegate to external tools via IPC
   (FreeSurfer, FSL, MRI scanners, custom scripts), not just MNE-CPP functions.
   A single graph can orchestrate C++, Python, and external CLI tools.
8. **Dynamic parameter adjustment**: The parameter tree allows attribute values
   to be recomputed at runtime via formulas referencing upstream outputs or
   stream metrics — enabling adaptive pipelines (e.g. auto-tuning
   regularisation based on real-time noise estimates).

**Data kinds flowing between nodes** (`MnaDataKind` enum):

| Kind | Description | Typical source |
|------|-------------|----------------|
| `FiffRaw` | Continuous raw M/EEG data | File or acquisition |
| `FiffEvoked` | Averaged evoked response | Averaging node |
| `FiffEpochs` | Epoched data | Epoching node |
| `FiffCov` | Noise or data covariance matrix | Covariance node |
| `Forward` | Forward operator (gain matrix + source space) | Forward model node |
| `Inverse` | Inverse operator | Inverse node |
| `SourceEstimate` | Source-space time series (STC) | Source estimation node |
| `Surface` | FreeSurfer mesh (vertices + triangles) | File input |
| `Volume` | MRI volume (.mgz / .nii) | File input |
| `Matrix` | Generic Eigen matrix (e.g. connectivity) | Any computation |
| `Scalar` | Single numeric value | Parameter or metric |
| `OnnxModel` | Trained ONNX model file | Training or file input |
| `FilePath` | Opaque file reference | Any file |
| `RealTimeStream` | Live data channel (e.g. MEG acquisition, LSL inlet, FIFF real-time server) | MNE Scan plugin or external device |
| `IpcEndpoint` | Inter-process communication handle (socket, pipe, shared-memory segment) | External tool (FreeSurfer, MRI scanner, custom process) |
| `Custom` | User-defined type (string tag) | Extension ops |

### Class design: `MnaFileRef`

```
struct MNASHARED_EXPORT MnaFileRef
{
    MnaFileRole role;           // What this file is: Raw, Forward, Inverse, Covariance, etc.
    QString     path;           // Relative POSIX path to the file
    QString     sha256;         // SHA-256 hex digest (verified on read, warn if mismatch)
    QString     format;         // File format hint: "fiff", "mgh", "stc", "json", etc.
    qint64      sizeBytes;      // File size in bytes (for integrity checking)

    // Packaging support (v2.2.0: reserved; v2.3.0+: .mnx can embed file content)
    bool        embedded;       // false = external file reference (default)
                                // true  = file content stored in `data` (future .mnx packaging)
    QByteArray  data;           // Raw file bytes when embedded=true; empty otherwise
                                // Writer ignores when embedded=false
                                // Reader ignores data key when embedded=false
};
```

### Class design: `MnaPort`

```
struct MNASHARED_EXPORT MnaPort
{
    QString     name;           // Port name (unique within a node, e.g. "raw_data", "covariance")
    MnaDataKind dataKind;       // What type of data: FiffRaw, Forward, SourceEstimate, Matrix, …
    MnaPortDir  direction;      // Input or Output

    // Connection (for input ports only — identifies the upstream source)
    QString     sourceNodeId;   // Which node produces this input? (empty → graph-level input)
    QString     sourcePortName; // Which output port on that node? (empty → graph-level input)

    // Real-time stream binding (used when dataKind == RealTimeStream)
    // When set, the port reads from / writes to a live data channel instead of a file.
    QString     streamProtocol; // "fiff-rt" (FIFF real-time server), "lsl" (LabStreamingLayer),
                                // "tcp", "shm" (shared memory), "" = not a stream
    QString     streamEndpoint; // Protocol-specific address:
                                //   fiff-rt: "hostname:port"
                                //   lsl:     "stream-name" (resolved via LSL)
                                //   tcp:     "hostname:port"
                                //   shm:     "segment-name"
    int         streamBufferMs; // Ring-buffer length in milliseconds (0 = unbounded)

    // Cached result reference (for output ports — populated after execution)
    QString     cachedResultPath; // Relative path to cached result file (empty → not yet computed)
    QString     cachedResultHash; // SHA-256 of cached result (for invalidation)
};
```

### Class design: `MnaNode` (replaces `MnaStep`)

```
struct MNASHARED_EXPORT MnaNode
{
    QString     id;             // Unique node identifier ("node-001", "compute_fwd_01")
    QString     opType;         // Operation type string — looked up in MnaOpRegistry
                                // Examples: "load_fiff_raw", "compute_covariance",
                                // "make_forward", "apply_inverse_dspm", "filter_bandpass",
                                // "compute_cmne", "train_lstm", "export_stc"
    QVariantMap attributes;     // Operation parameters (typed key-value)
                                // e.g. {"lambda2": 0.1111, "method": "dSPM", "loose": 0.2}

    QList<MnaPort> inputs;      // Input ports — each references an upstream node's output port
    QList<MnaPort> outputs;     // Output ports — named, typed slots for this node's results

    // Execution mode
    MnaNodeExecMode execMode;   // Batch   — runs once on static inputs (default, file-based)
                                // Stream  — runs continuously on real-time data (MNE Scan mode)
                                // Ipc     — delegates to an external process via pipes/sockets
                                // Script  — inline code executed via interpreter (Python, shell, R, …)

    // IPC configuration (used when execMode == Ipc)
    QString     ipcCommand;     // Executable path or command line (e.g. "recon-all", "mri_convert")
    QStringList ipcArgs;        // Command-line arguments (may contain {{port_name}} placeholders
                                // that are substituted with actual file paths at execution time)
    QString     ipcWorkDir;     // Working directory for the external process
    QString     ipcTransport;   // IPC transport: "stdio" (stdin/stdout pipes), "tcp", "shm", "file"
                                // "stdio" is the default — the node writes input data to the
                                // process's stdin and reads output from stdout.
                                // "file" mode: inputs written to temp files whose paths are
                                // substituted into ipcArgs; outputs read from declared result paths.

    // Execution metadata
    QString     toolVersion;    // Version of the tool that last executed this node
    QDateTime   executedAt;     // When this node was last run (empty → not yet executed)
    bool        dirty;          // true if attributes changed since last execution
                                // (or an upstream node is dirty) → needs re-execution
};
```

### Real-Time Streaming — MNA as MNE Scan Configuration

An MNA/MNX graph is not limited to offline batch processing of files on disk.
When one or more ports carry the `RealTimeStream` data kind, the graph describes
a **live processing pipeline** that can be loaded directly by **MNE Scan** (the
real-time acquisition and processing application in MNE-CPP).

**How it works:**
1. A graph-level input port with `dataKind: RealTimeStream` declares a live
   data source (e.g. an MEG acquisition, an EEG amplifier via LSL, or a FIFF
   real-time server).
2. Downstream nodes whose `execMode` is `Stream` operate on continuous chunks
   rather than complete files. They receive data from a ring buffer, process
   the current window, and push results to their output ports in real time.
3. Nodes with `execMode: Batch` in a mixed graph are executed once at startup
   (e.g. loading a forward solution from disk) and their outputs remain
   available as static inputs to streaming nodes.
4. An output port with `RealTimeStream` produces a continuous data channel
   that can feed a display widget, a BCI controller, or a network socket.

**Supported stream protocols** (`MnaPort::streamProtocol`):

| Protocol | Transport | Use case |
|----------|-----------|----------|
| `fiff-rt` | TCP (FIFF real-time protocol) | Elekta/MEGIN acquisition, mne_rt_server |
| `lsl` | Network (LabStreamingLayer) | Cross-application streaming (EEG, markers) |
| `tcp` | Raw TCP socket | Generic numeric streams |
| `shm` | POSIX shared memory / Windows named mapping | Ultra-low-latency on-device transfer |

This means an `.mna` file can serve as a **configuration file for MNE Scan**: it
declares which acquisition source to connect to, which processing steps to apply
in real time (filtering, artifact rejection, source localisation), and where to
route the results (display, file, network). MNE Scan loads the graph, resolves
stream endpoints, and runs the pipeline.

### IPC / External Process Nodes

Not every node in the graph needs to be an MNE-CPP function. When `execMode` is
`Ipc`, the node delegates execution to an **external process** via inter-process
communication. This supports tools like FreeSurfer, FSL, SPM, custom MRI
acquisition scripts, or any command-line utility.

**Execution flow for an IPC node:**
1. The executor prepares inputs: upstream results are written to temporary files
   (or piped via stdin, depending on `ipcTransport`).
2. `ipcCommand` is launched with `ipcArgs`, where `{{placeholder}}` tokens in
   the argument list are replaced with actual file paths or stream addresses.
3. The executor monitors the child process (stdout for progress, stderr for
   errors, exit code for success/failure).
4. On completion, the executor reads declared output files (or stdout) and
   populates the node's output ports in the execution context.

**Example: FreeSurfer reconstruction node**
```json

  "ipc_transport": "file",
  "attributes": { "parallel_jobs": 4 },
  "inputs": [
    { "name": "t1_volume", "data_kind": "Volume",
      "source_node": "load_mri_01", "source_port": "volume" }

  "outputs": [
```

### Parameter Tree with Dynamic Bindings

Static parameters on nodes (the `attributes` map) are sufficient for batch
pipelines. But in real-time and adaptive scenarios, parameters need to change
**during** execution — for example, adjusting a regularisation parameter based
on the current noise level, or switching a BCI classifier threshold based on
the subject's performance.

The **`MnaParamTree`** is a hierarchical parameter store attached to the graph.
Each parameter has a path (e.g. `"apply_inv_01/lambda2"`) and an optional
**binding** — a formula or operator that recomputes the parameter value whenever
a trigger condition is met.

**Key concepts:**
- **Parameter path**: `"<nodeId>/<attributeKey>"` — uniquely identifies any
  attribute on any node in the graph.
- **Binding**: A formula (simple arithmetic expression or lookup) that references
  other parameters or node outputs. Evaluated at runtime.
- **Trigger**: When to re-evaluate — `on_change` (when a referenced source
  changes), `periodic` (every N milliseconds), or `manual` (explicit API call).
- **Operators**: Built-in functions — `scale(x, factor)`, `clamp(x, min, max)`,
  `threshold(x, value, above, below)`, `lerp(a, b, t)`, `moving_avg(x, window)`,
  `ref(path)` (read another parameter's value).

### Class design: `MnaParamBinding`

```
struct MNASHARED_EXPORT MnaParamBinding
{
    QString    targetPath;      // Parameter to control: "nodeId/attrKey"
    QString    expression;      // Formula string: "clamp(ref('noise_est_01/snr') * 0.1, 0.01, 1.0)"
    QString    trigger;         // "on_change", "periodic", "manual"
    int        periodMs;        // Evaluation period when trigger == "periodic" (ignored otherwise)
    QStringList dependencies;   // Paths this binding reads from (auto-derived from expression,
                                // but can be explicit for complex cases)
};
```

### Class design: `MnaParamTree`

**JSON representation** in the `.mna` file:
```json
{
  "param_tree": {
    "bindings": [
      {
        "target": "apply_inv_01/lambda2",
        "expression": "clamp(1.0 / (ref('noise_est_01/snr') + 1.0), 0.01, 1.0)",
        "trigger": "on_change",
        "dependencies": ["noise_est_01/snr"]
      },
      {
        "target": "filter_01/highpass_hz",
        "expression": "threshold(ref('artifact_det_01/line_noise_power'), 0.5, 1.0, 0.1)",
        "trigger": "periodic",
        "period_ms": 5000
      }
    ]
  }
}
```

In the first binding, `lambda2` is automatically adjusted whenever the upstream
noise estimation node produces a new SNR value — higher SNR → lower
regularisation. In the second, the highpass filter cutoff is raised from 0.1 Hz
to 1.0 Hz when line-noise power exceeds a threshold, checked every 5 seconds.

This parameter tree architecture is inspired by the adaptive parameter control
concepts described in the **Röll & Dinh MR patent** (see TASK 17 bibliography),
which formalized the idea of formula-driven parameter adjustment in real-time
neuroimaging pipelines.

### Script Nodes — Inline Code as First-Class Graph Operations

IPC nodes (above) delegate work to an **external executable**. But many
scientific workflows require small, ad-hoc analysis steps that are most naturally
expressed as a short Python, R, or shell script — not a compiled binary. Forcing
users to save these snippets as separate `.py` files and wire them through
`ipc_command` creates unnecessary friction and breaks the self-contained nature
of the `.mna` file.

**Script nodes** solve this by embedding the source code **directly inside the
node definition**. The executor writes the code to a temporary file, invokes
the appropriate interpreter, and captures the results — exactly like
`PythonRunner::runCode()` (TASK 16) but generalized to any language.

A new execution mode `MnaNodeExecMode::Script` is added alongside Batch,
Stream, and Ipc. When the executor encounters a Script node, it:

1. Reads the `script` struct from the node definition.
2. Writes `script.code` to a temporary file with the correct extension
   (`.py`, `.sh`, `.R`, `.m`, …).
3. Substitutes `{{port_name}}` placeholders in the code with actual file paths
   or values from upstream node outputs (same mechanism as IPC args).
4. Launches the interpreter: `script.interpreter` with `script.interpreterArgs`
   plus the temp file path.
5. Captures stdout/stderr, parses progress lines, and populates output ports.
6. The temporary file is deleted after execution (unless `script.keepTempFile`
   is true, useful for debugging).

**Why this matters:**
- A single `.mna` file becomes a **complete, portable experiment definition**:
  data references, processing graph, parameter tree, *and* custom analysis code.
- Reviewers can inspect exactly what script ran — the code is versioned with the
  pipeline, not hidden in an external file that may have changed since execution.
- For real-time graphs, script nodes with short execution times can be called per
  chunk (e.g. a 10-line Python snippet that computes a custom metric each second).

### Verification & Provenance — The Scientific Audit Trail

Scientists are obsessed with the *"Why"* behind a result. If a format can prove
exactly which filter coefficients and regularisation $\lambda$ were used at
$t=180\text{s}$ of a live session, it wins. Every MNA node therefore carries an
**`MnaVerification`** record — a structured audit trail that captures what
happened, what was expected, and whether those expectations held.

Verification serves three purposes:

1. **Provenance** — a machine-readable log of every parameter, software version,
   and input hash that contributed to a node's output. Enough to reproduce or
   challenge the result years later.
2. **Runtime assertions** — declarative checks that fire *during* execution and
   either warn or abort when data violates domain assumptions (e.g. "covariance
   matrix must be positive-definite", "epoch count ≥ 40").
3. **Explanation** — a human-readable rationale attached to each node: *why*
   this operation was chosen, *what* the expected effect is, and *how* to
   interpret failures. This turns the `.mna` file into a literate-programming
   artefact that documents intent alongside computation.

**Verification lifecycle:**

| Phase | What happens | Stored where |
|-------|-------------|--------------|
| **Authoring** | User writes `checks[]` (pre/post-conditions) and `explanation` on the node | `.mna` file |
| **Pre-execution** | Executor evaluates pre-conditions; if a `severity: "error"` check fails, execution aborts with a clear message | `verification.preResults[]` |
| **Post-execution** | Executor evaluates post-conditions against actual outputs; warnings are logged, errors abort downstream execution | `verification.postResults[]` |
| **Snapshot** | Executor writes complete provenance: resolved input hashes, attribute values at execution time, software versions, wall-clock time, environment variables | `verification.snapshot` |

### Class design: `MnaVerificationCheck`

```
struct MNASHARED_EXPORT MnaVerificationCheck

    QString     description;    // Human-readable: "Covariance matrix must be positive-definite"

                                //   "rank(covariance) > 0"
```

### Architecture: Single Schema, Dual Serializer

The `.mna` and `.mnx` formats share **exactly the same internal data model**
(key names, hierarchy, value types). The only difference is the serializer:

| Aspect | `.mna` (JSON) | `.mnx` (CBOR) |
|--------|---------------|---------------|
| Encoding | UTF-8 text | Binary (RFC 8949) |
| Editability | Human-readable; open in any text editor or VS Code | Requires hex editor or MNE-CPP app |
| Numeric storage | Strings ("3.14") | Native 4/8-byte IEEE floats/ints |
| Binary data | Base64-encoded (33% overhead) | Native byte-string type (zero overhead) |
| Primary use | Metadata, parameters, small projects, human review | Large projects, embedded data payloads, interchange |

The code uses a single `MnaProject` data model. `MnaIO::write()` checks the
file extension and toggles between `QJsonDocument` serialization (`.mna`) and
`QCborValue` serialization (`.mnx`). `MnaIO::read()` does the same in reverse.
This means every feature works identically for both formats.

### `.mnx` Magic Header

To prevent confusion with arbitrary CBOR files, `.mnx` files start with a
4-byte magic identifier before the CBOR payload:

```
Bytes 0-3: "MNX1"   (ASCII magic string + format version digit)
Bytes 4-N: CBOR payload (QCborValue serialization of the MnaProject)
```

`MnaIO::read()` checks the magic before parsing. If the magic is missing or
unrecognized, it returns an error with a clear message ("Not a valid .mnx file
or unsupported version"). The version digit (`1`) allows future schema changes
without breaking old readers.

### `.mnx` as a Packaging Format (Roadmap)

In v2.2.0, `.mnx` stores the same manifest as `.mna` — file references with
relative paths pointing to external files on disk.

In a future release (v2.3.0+), `.mnx` will evolve into a true **packaging
container** (analogous to MP4 for multimedia) that can **embed** the actual
file contents directly:

```
"files": [
  {
    "role": "raw",
    "path": "sub-01_meg.fif",
    "embedded": true,
    "data": <CBOR byte-string, native bytes of the .fif file>,
    "sha256": "a1b2c3...",
    "size_bytes": 1073741824
  }
]
```

CBOR's native byte-string type stores the raw bytes at zero overhead (no Base64),
making this efficient even for large FIFF files or MRI volumes. A single `.mnx`
file would then contain the entire analysis — raw data, MRI, surfaces, forward
model, inverse, source estimates — fully self-contained and portable, like
handing someone a single USB stick instead of a folder tree.

**v2.2.0 scope**: The `MnaFileRef` struct gains an `embedded` boolean field
(default `false`) and an optional `QByteArray data` field. The writer ignores
`data` when `embedded` is false. The reader ignores the `data` key if present
but `embedded` is false. This reserves the field for v2.3.0 without adding
complexity now.

### Versioning

The `mna_version` field at the root of every `.mna`/`.mnx` file tracks the
schema version. The reader must:
1. Check `mna_version` before parsing.
2. If major version differs (e.g. "2.x" vs "1.x"), refuse to load and report
   the version mismatch.
3. If minor version is newer (e.g. "1.2" vs "1.0"), load gracefully and ignore
   unknown keys (forward compatibility).

v2.2.0 ships with schema version `"1.0"`.

### Acceptance criteria

**Serialization & I/O:**
- `MnaProject::read("test.mna")` round-trips a project losslessly.
- `MnaProject::read("test.mnx")` round-trips with CBOR.
- `.mnx` magic header is validated; reading a non-MNX binary file returns a
  clear error (not a crash or silent corruption).
- SHA-256 hashes are verified on read (warn if mismatch, don't fail).
- All paths stored as relative POSIX paths (forward slashes).
- `mna_version` is written and checked on read; version mismatch handling works.
- `MnaNode` attribute dictionaries round-trip with correct types (doubles stay
  doubles, strings stay strings, ints stay ints).

**Graph model:**
- **Graph round-trip**: A graph with 5+ nodes, typed ports, edges between nodes,
  and graph-level inputs/outputs serializes to `.mna` (JSON) and `.mnx` (CBOR)
  and reads back identically. All `source_node`/`source_port` references are
  preserved.
- **DAG validation**: `MnaGraph::validate()` detects and reports: (a) cycles,
  (b) disconnected required input ports, (c) data-kind mismatches between
  connected ports, (d) missing required attributes, (e) unknown op types.
- **Topological sort**: `MnaGraph::topologicalSort()` returns a valid execution
  order. For a diamond graph (A→B, A→C, B→D, C→D), D comes after both B and C.
- **Schema validation**: `MnaOpSchema::validate(node)` catches: wrong number of
  ports, wrong data kinds, missing required attributes, wrong attribute types.
- **Op registry**: Built-in ops are registered at init. `MnaOpRegistry::hasOp()`
  returns true for standard ops. Custom ops can be registered and looked up.

**Graph execution:**
- **Full execution**: `MnaGraphExecutor::execute()` traverses a 3-node pipeline
  (covariance → forward → inverse) in correct order, passes intermediate
  results between nodes, and returns final outputs.
- **Incremental execution**: Marking one node dirty and calling
  `executeIncremental()` re-runs only that node and its downstream dependents;
  upstream cached results are reused.
- **Dirty propagation**: Changing an attribute on a node marks it and all
  downstream nodes dirty. Upstream nodes stay clean.

**Real-time streaming:**
- **Stream port round-trip**: A port with `dataKind: RealTimeStream`,
  `streamProtocol: "lsl"`, and `streamEndpoint: "eeg-stream-01"` round-trips
  through `.mna` (JSON) and `.mnx` (CBOR) with all stream fields preserved.
- **Mixed graph validation**: A graph with both `Batch` and `Stream` nodes
  validates correctly — batch nodes are upstream of stream nodes; stream nodes
  may not feed batch nodes (unless through a buffering intermediary).
- **MNE Scan config**: A `.mna` file with a `fiff-rt` stream input can be loaded
  and the stream endpoint and buffer size extracted programmatically.

**IPC nodes:**
- **IPC node round-trip**: An `MnaNode` with `execMode: Ipc`, `ipcCommand`,
  `ipcArgs` (with `{{placeholder}}` tokens), `ipcTransport`, and `ipcWorkDir`
  round-trips losslessly through `.mna` and `.mnx`.
- **Placeholder substitution**: `ipcArgs` containing `{{raw_data}}` are correctly
  expanded to the actual file path at execution time.
- **IPC validation**: `MnaGraph::validate()` verifies that IPC nodes have a
  non-empty `ipcCommand` and a valid `ipcTransport`.

**Parameter tree:**
- **Binding round-trip**: A `MnaParamTree` with 2+ bindings (different triggers)
  serializes to JSON / CBOR and reads back identically — expressions, triggers,
  dependencies all preserved.
- **Expression evaluation**: `evaluateExpression("clamp(3.0 * 0.5, 0.1, 1.0)",
  ctx)` returns `1.0`. `ref('nodeId/attr')` correctly reads from the context.
- **On-change trigger**: Changing an upstream output causes dependent bindings
  to fire and update target attributes. The target node is marked dirty.
- **Periodic trigger**: Bindings with `trigger: "periodic"` are evaluated at the
  specified interval during stream execution.

**Data-driven test:**
- Build a realistic project referencing actual test data from
  `mne-cpp-test-data/` (raw FIFF, forward solution, covariance, STC), construct
  a graph with 3+ nodes and realistic parameters, write to `.mna`, read back,
  verify every field including graph topology and port connections. Repeat for
  `.mnx`.

**Edge cases:**
- Empty project (no subjects, empty graph).
- Project with 5 subjects × 3 sessions × 2 runs.
- File ref with non-ASCII path characters.
- Graph with a single node (no edges).
- Graph with parallel branches that merge (diamond pattern).
- Node with optional input port left unconnected.

**Coverage:** ≥75% line coverage for all `mna` library source files.

## TASK 2 — New Library: `ml` (Machine Learning)

### Goal
Create the `ml` library (`src/libraries/ml/`) providing ONNX Runtime integration
for inference, lightweight model wrappers for loading and running pre-trained
models, feature preprocessing pipelines, and a Python training bridge
(`MLTrainer`) that invokes external training scripts via `PythonRunner`.

### Why
ML-based artifact rejection, source imaging, and brain-computer interfaces are
becoming standard in neuroscience. Today mne-cpp has zero ML capability. This
library bridges that gap by letting users (a) run pre-trained ONNX models for
inference and (b) launch model training in Python from C++ via `MLTrainer`,
then load the resulting `.onnx` models back for inference. The training-in-Python
/ inference-in-C++ split keeps the library lean while leveraging the full Python
ML ecosystem (PyTorch, scikit-learn, etc.).

### Class design: `MlModel` (abstract base)

```
class MLSHARED_EXPORT MlModel
{
public:
    typedef QSharedPointer<MlModel> SPtr;

    virtual ~MlModel() = default;

    // Inference: input tensor → output tensor
    virtual MlTensor predict(const MlTensor& input) const = 0;

    // Persistence
    virtual bool save(const QString& path) const = 0;    // .onnx or .json
    virtual bool load(const QString& path) = 0;

    // Metadata
    virtual QString modelType() const = 0;       // "onnx", "linear", "logistic"
    virtual MlTaskType taskType() const = 0;
};
```

**Note**: There is no `train()` method on the model classes. All training is
handled via Python scripts launched through `MLTrainer` (see below). This keeps
the C++ class hierarchy clean: models are inference-only containers with
save/load serialisation.

### `MlOnnxModel` specifics
- `load()` creates an `Ort::Session` from the `.onnx` file.
- `predict()` maps `MlTensor` to `Ort::Value`, runs session, maps back.
- `save()` is a no-op for externally-trained models (the file already exists).

### `MlLinearModel` specifics
- Supports `MlTaskType::Classification` (logistic regression with sigmoid) and
  `MlTaskType::Regression` (ridge regression).
- `load()` reads weights + bias from JSON or ONNX format.
- `predict()` applies the linear transformation `y = W*x + b` (with optional
  sigmoid for classification).
- `save()` exports as lightweight JSON (weights matrix + bias vector) or ONNX.
- For training a linear model, use `MLTrainer` with a scikit-learn script
  that exports weights to JSON/ONNX.

### `MLTrainer` design
```
class MLSHARED_EXPORT MLTrainer
{
public:
    MLTrainer();
    explicit MLTrainer(const UTILSLIB::PythonRunnerConfig& config);

    // Access the underlying PythonRunner for callback/config changes
    UTILSLIB::PythonRunner& runner();

    // Run a training script (auto-manages venv if configured)
    UTILSLIB::PythonRunnerResult run(const QString& scriptPath,
                                     const QStringList& args = {});

    // Check that required Python packages are importable
    // Returns list of packages that could NOT be imported (empty = all OK)
    QStringList checkPrerequisites(const QStringList& packages) const;
};
```

`MLTrainer` is the single entry point for all Python-based training from C++.
It is used by:
- `InvCMNE::trainLstm()` for CMNE LSTM training (TASK 9)
- Any future training workflows (custom classifiers, deep nets, etc.)
- CLI tools like `mne_compute_cmne --train`

## TASK 3 — New Library: `sts` (Statistical Testing)

### Goal
Create the `sts` library (`src/libraries/sts/`) providing non-parametric
cluster permutation tests, parametric t/F-tests, and multiple-comparison
corrections (FDR, Bonferroni). This is the #1 most-used MNE-Python feature that
mne-cpp completely lacks (see gap-analysis §8).

The library directory is named `sts` (≤5 letters, matching the naming convention)
but class names keep the more readable `Stats` prefix (e.g. `StatsCluster`,
`StatsTtest`). Namespace is `STSLIB`, export macro is `STSSHARED_EXPORT`.

### Why
Every neuroscience paper requires statistical testing of M/EEG contrasts.
Cluster-based permutation testing (Maris & Oostenveld, 2007) is the gold standard
for spatio-temporal data because it controls family-wise error rate without
assuming independence of time/space points. Without this library, mne-cpp users
must export data to Python/R for statistics.

### Files to create

| File | Purpose |
|------|---------|
| `sts_global.h` | Export macro `STSSHARED_EXPORT` |
| `sts_global.cpp` | Build-info stamp |
| `sts_types.h` | Enums: `StatsTailType` (Left, Right, Both), `StatsCorrection` (None, Bonferroni, Fdr, ClusterPermutation) |
| `sts_ttest.h/.cpp` | `StatsTtest` — one-sample, paired, independent t-tests (parametric); returns t-statistics + p-values as Eigen matrices |
| `sts_ftest.h/.cpp` | `StatsFtest` — one-way and repeated-measures F-test (ANOVA); returns F-statistics + p-values |
| `sts_correction.h/.cpp` | `StatsMcCorrection` — FDR (Benjamini-Hochberg), Bonferroni, Holm-Bonferroni; operates on p-value matrices |
| `sts_cluster.h/.cpp` | `StatsCluster` — spatio-temporal cluster permutation test (the core feature) |

### Class design: `StatsCluster`

This is the most important class. It implements the Maris & Oostenveld (2007)
cluster-based permutation test for 2D data (channels × time or vertices × time).

```

{

    QVector<double>  vecClusterStats;  // Sum-t for each significant cluster

{
public:
    // Main entry point: cluster permutation t-test between two conditions
    // matA: condition A data (n_observations × n_channels × n_times), stored
    //       as a QVector of Eigen::MatrixXd (one matrix per observation)
    // matB: condition B data (same shape)

    //            from StatsAdjacency

        const QVector<Eigen::MatrixXd>& dataB,
        const Eigen::SparseMatrix<int>& adjacency,
        int    nPermutations = 1024,
        double clusterAlpha  = 0.05,
        double pThreshold    = 0.05,
        StatsTailType tail   = StatsTailType::Both);

    // Compute t-statistic map from two groups
    static Eigen::MatrixXd computeTMap(
        const QVector<Eigen::MatrixXd>& dataA,
        const QVector<Eigen::MatrixXd>& dataB);

    // Find connected clusters above threshold in t-map using adjacency
    static QPair<Eigen::MatrixXi, QVector<double>> findClusters(
        const Eigen::MatrixXd& tMap,
        double threshold,
        const Eigen::SparseMatrix<int>& adjacency,
        StatsTailType tail);

    static double permuteOnce(
        const QVector<Eigen::MatrixXd>& allData,
        int nA,
        const Eigen::SparseMatrix<int>& adjacency,
        double threshold,
        StatsTailType tail);

```

### Algorithm (for the implementing agent)

1. **Observed t-map**: Compute paired or independent t-statistic at each
   (channel, time) point from `dataA` vs `dataB`.
2. **Threshold**: Find all (channel, time) points where |t| > threshold
   (derived from `clusterAlpha` via inverse t-distribution or direct threshold).
3. **Cluster formation**: Use breadth-first-search on the adjacency graph ⊗ time
   adjacency (consecutive time points are always adjacent) to find connected
   components of supra-threshold points.
4. **Cluster statistic**: Sum the t-values within each cluster.
5. **Permutation distribution**: Repeat `nPermutations` times:
   a. Randomly reassign observations to groups A/B.
   b. Compute t-map → clusters → take the maximum cluster sum-t.
   c. Store in null distribution.
6. **p-values**: For each observed cluster, p = (# null ≥ observed) / nPermutations.
7. **Parallelise** permutations using `QtConcurrent::mappedReduced` or
   `#pragma omp parallel for`.

## TASK 5 — Multitaper PSD & TFR (DPSS)

### Goal
Add multitaper spectral estimation to `src/libraries/dsp/` — both static PSD and
time-frequency resolved (TFR). This fills a high-priority gap: Thomson's

### Why
Multitaper PSD is the standard in neuroscience for spectral analysis of M/EEG
data. It provides optimal bias-variance tradeoff controlled by the half-bandwidth
parameter. The existing `WelchPsd` class handles simple periodogram averaging but
does not support DPSS tapers or time-frequency resolution.

### Files to create

|------|---------|
| `dpss.h/.cpp` | `Dpss` — compute Discrete Prolate Spheroidal Sequences (Slepian tapers). Static method: `Dpss::compute(int N, double halfBandwidth, int nTapers)` returns `Eigen::MatrixXd` (nTapers × N). Algorithm: solve the tridiagonal eigenvalue problem from Slepian (1978). |
| `multitaper_psd.h/.cpp` | `MultitaperPsd` — apply DPSS tapers, FFT each taper, average. Interface mirrors `WelchPsd`: static `compute(matData, sfreq, halfBandwidth, nTapers)` returns `MultitaperPsdResult { matPsd, vecFreqs }`. |
| `multitaper_tfr.h/.cpp` | `MultitaperTfr` — sliding-window multitaper: segment data into overlapping windows, apply DPSS within each window. Returns 3D result (channels × freqs × times). Store as `MultitaperTfrResult { QVector<Eigen::MatrixXd> tfrData, vecFreqs, vecTimes }`. |

### Modify
- `src/libraries/dsp/CMakeLists.txt`: add new files to SOURCES/HEADERS.

### Algorithm: DPSS computation
The DPSS are the eigenvectors of the N×N tridiagonal matrix:
```
T(i,i) = ((N-1-2i)/2)^2 * cos(2π * W)     for i = 0..N-1
T(i,i+1) = T(i+1,i) = i*(N-i)/2
```
where W = halfBandwidth/N. Use Eigen's `SelfAdjointEigenSolver` on the
tridiagonal matrix. Return the top `nTapers` eigenvectors (highest eigenvalues).
Eigenvalues (concentration ratios) should also be returned for adaptive weighting.

### Acceptance criteria
- **DPSS validation**: Tapers for N=256, W=4 match SciPy `scipy.signal.windows.dpss()`
  to <1e-10 relative error (pre-compute SciPy reference values and embed as
  test constants). Eigenvalues (concentration ratios) match to <1e-8.
- **Data-driven PSD test**: Load real MEG data from `mne-cpp-test-data/`,
  compute multitaper PSD, verify alpha peak (~10 Hz) is visible in posterior
  channels (peak frequency within 8–13 Hz band).
- **Synthetic PSD test**: Generate 10 Hz + 40 Hz sinusoids in white noise

- **TFR test**: Generate a linear chirp (10→50 Hz over 2s). Multitaper TFR

```
        bool extendedMode = true,                // true = extended (sub+super Gaussian)
        unsigned int seed = 0);                  // 0 = random seed

    // Updated every block using kurtosis estimate
    static Eigen::VectorXd estimateSignVector(const Eigen::MatrixXd& matSources);
```

### Algorithm (for the implementing agent)
1. **Whitening**: PCA → whitening transform (same as FastICA preprocessing).
2. **Initialize** weights W as identity (or small random perturbation).
3. **For each iteration**:
   a. Compute sources: `S = W * X_white`.
   b. Every N samples, estimate kurtosis of each source to get sign vector
      `k`: `k(i) = +1` if excess kurtosis > 0 (super-Gaussian),
      `k(i) = -1` if < 0 (sub-Gaussian).
   c. Compute natural gradient update:
      `dW = (I - k * tanh(S) * S^T - S * S^T) * W`
      (the `k` vector selects the nonlinearity per component).
   d. `W = W + learningRate * dW`.
   e. Check convergence: `max(abs(dW)) < tolerance`.
4. **De-whiten** to get mixing/unmixing in original sensor space.

```
    // nIterations: max EM iterations
    // tolerance: convergence criterion on marginal log-likelihood change
    // gammaPruneThresh: sources with γ < threshold are pruned from active set
    static InvGammaMapResult compute(
        const Eigen::MatrixXd& matGain,
        const Eigen::MatrixXd& matData,
        const Eigen::MatrixXd& matNoiseCov,
        double alpha = 1.0,
        double beta  = 1.0,
        int nIterations = 100,
        double tolerance = 1e-6,
        double gammaPruneThresh = 1e-10);
};

}
```

### Algorithm: MxNE (for the implementing agent)
MxNE minimizes: `||M - G*X||²_F + α * Σ_i ||X_i||_2`
where `X_i` is the i-th source's time course (row of X), and `||·||_2` is the
L2 (Euclidean) norm of that row. This is a group-lasso / L21-norm penalty.

Solve via **Iterative Reweighted Least Squares (IRLS)**:
1. Initialize weights `w_i = 1` for all sources.
2. For each iteration:
   a. Form diagonal weight matrix `W = diag(w_1, ..., w_n)`.
   b. Solve weighted ridge: `X = (G^T G + α W^{-2})^{-1} G^T M`.
   c. Update weights: `w_i = max(||X_i||_2, ε)` (ε = 1e-10 for stability).
   d. Prune sources where `w_i < ε_prune` (active set strategy for speed).
   e. Check convergence: `max |w_new - w_old| < tolerance`.
3. Return sparse `X` with only active sources.

### Algorithm: Gamma-MAP (for the implementing agent)
Gamma-MAP places a Gamma hyperprior on source variances:
1. Initialize source variances `γ_i = 1`.
2. E-step: Compute posterior mean `X = Σ_post G^T Σ_noise^{-1} M` where
   `Σ_post = (G^T Σ_noise^{-1} G + diag(1/γ))^{-1}`.
3. M-step: Update `γ_i = ||X_i||²_2 / T` where T is number of time points.
4. Prune sources where `γ_i < threshold`.
5. Repeat until convergence.

### Goal
Implement CMNE in `src/libraries/inv/minimum_norm/`. CMNE uses a
spatiotemporal LSTM network to predict the next source estimate from a
sequence of past estimates, then multiplies (element-wise) this prediction
with the current dSPM estimate to produce a context-corrected source map.
Unlike standard MNE/dSPM which operates sample-by-sample, CMNE exploits the
temporal structure of brain activity to improve spatial fidelity.

The ONNX Runtime integration from the `ml` library (TASK 2) provides the LSTM
inference engine; training is done offline in Python (reference repo above).

### Mathematical formulation (from the paper)

**Standard dSPM (time-invariant, per-sample)**:

1. Whitened data: `ỹ_t = C_n^{-1/2} y_t`
2. Whitened gain: `G̃ = C_n^{-1/2} G`
3. MNE kernel: `K = C_R G̃^T (G̃ C_R G̃^T + λ²I)^{-1}`
4. dSPM normalisation: `K_dSPM = W_dSPM K` where
   `W_dSPM(i,i) = 1 / sqrt(diag_i(K C_n K^T))`
5. Estimate: `x̂_t = K_dSPM ỹ_t`
6. Rectify + z-score: `q̂_t(i) = (|x̂_t(i)| - μ) / σ`

**CMNE correction (time-dependent, recursive)**:

7. For `t < k` (look-back window): `b_t = q̂_t` (identity — no correction)
8. For `t ≥ k`:
   a. Feed `{b_{t-k}, ..., b_{t-1}}` into the LSTM network.
   b. LSTM output → `|LSTM(...)| / max(|LSTM(...)|)` → diagonal of `W_t^CMNE`.
   c. CMNE estimate: `b_t = W_t^CMNE q̂_t` (element-wise product).
   d. `b_t` replaces `q̂_t` in the sliding window (Markov chain).

This forms a recursive Markov chain where the CMNE estimate itself is used as
input to the LSTM for subsequent predictions (Eq. 13 in the paper).

**Hyperparameters** (from the paper's cross-validation):
- `k = 80` — LSTM look-back (number of past time steps)
- `d = 1280` — number of hidden units in the LSTM layer
- Network: single LSTM layer → single Dense(n_sources, linear) output layer
- Loss: MSE, Optimiser: Adam
- Training: 85% epochs train / 15% validation, minibatch=30, 30 windows/batch

**Performance metrics** (from the paper):
- Peak localisation error: `PE = ||r_i - r_j||_2`
- Spatial dispersion: `SD = Σ_k d_jk |x̂_k| / Σ_k |x̂_k|`
- Source-space SNR: `SNR = σ²(active, signal) / σ²(active, noise)`

### Algorithm (for the implementing agent)

Reference: `examples/ex_cmne.py` in the `chdinh/cmne` repository.

1. **Compute static dSPM kernel** (Eqs. 5–8):
   a. Whiten gain matrix: `G̃ = C_n^{-1/2} G`.
   b. MNE kernel: `K = C_R G̃^T (G̃ C_R G̃^T + λ²I)^{-1}`.
   c. dSPM normalisation: for each source `i`,
      `w_i = 1.0 / sqrt((K C_n K^T)(i,i))`. Form diagonal `W_dSPM`.
   d. `K_dSPM = W_dSPM * K`.
   Validate dimensions: K_dSPM is `(n_sources × n_channels)`.
   Cross-check with MNE-Python: `mne.minimum_norm.make_inverse_operator()`
   + `prepare_inverse_operator()` compute the same kernel. MNE-SVN C:
   `mne_inverse_util.c` → `mne_prepare_inverse_operator()`.

2. **Apply dSPM to all time samples**:
   `x̂ = K_dSPM * ỹ` → matrix `(n_sources × n_times)`.

3. **Rectify + z-score** (Eq. 9):
   For each source `i`: `q̂(i,:) = (|x̂(i,:)| - mean(|x̂(i,:)|)) / std(|x̂(i,:)|)`.
   Reference: `cmne.standardize()` in `cmne/data.py`.

4. **LSTM prediction loop** (Eqs. 10–13, Markov chain):
   a. Allocate result matrix `b` of shape `(n_sources × n_times)`.
   b. Copy first `k` columns from `q̂` into `b` (no correction for `t < k`).
   c. For `t = k` to `n_times - 1`:
      - Extract window: `input = b[:, t-k : t]` transposed to `(k, n_sources)`.
      - Run ONNX inference: `lstm_output = onnx_session.run(input)` → `(n_sources,)`.
      - Normalise: `w = |lstm_output| / max(|lstm_output|)`.
      - CMNE estimate: `b[:, t] = w .* q̂[:, t]` (element-wise product).
   Reference: the for-loop in `examples/ex_cmne.py` lines `stc_posterior = stc_sens[i+lb,:] * stc_predict`.

5. **Output**: Return both the corrected `b` and the uncorrected `q̂` as
   `InvSourceEstimate` objects.

## TASK 14 — Ledoit-Wolf Shrinkage Covariance (Bonus)

### Goal
Add the Ledoit-Wolf optimal shrinkage estimator to the `sts` library
(`src/libraries/sts/sts_cov_estimators.h/.cpp`), enhancing covariance
estimation for rank-deficient M/EEG data (e.g. after SSS/tSSS).

### Why
The current empirical covariance + Tikhonov regularisation requires manual tuning.
Ledoit-Wolf automatically computes the optimal shrinkage coefficient toward a
structured target (scaled identity), producing better-conditioned covariance
matrices for inverse solutions without user parameters.

### Where to add
The implementation lives in `sts_cov_estimators.h/.cpp` alongside the other
statistical utilities in the `sts` library (TASK 3):

```
// In sts_cov_estimators.h
namespace STSLIB {

class STSSHARED_EXPORT StsCovEstimators
{
public:
    // Ledoit-Wolf shrinkage estimator
    // matData: n_channels × n_samples (zero-mean)
    // Returns: shrunk covariance (n_channels × n_channels) and shrinkage coefficient alpha
    static QPair<Eigen::MatrixXd, double> ledoitWolf(const Eigen::MatrixXd& matData);
};

} // namespace STSLIB
```

### Algorithm
Analytical formula from Ledoit & Wolf (2004):
1. Compute sample covariance `S = X X^T / (n-1)`.
2. Target `F = trace(S)/p * I` (scaled identity).
3. Compute optimal shrinkage `α` using the Ledoit-Wolf formula
   (involves computing the sum of squared off-diagonal elements and a
   bias correction term proportional to 1/n).
4. Return `α*F + (1-α)*S`.

### Acceptance criteria
- **Data-driven rank test**: Load real noise covariance from
  `mne-cpp-test-data/` (has rank < n_channels due to SSS). Apply Ledoit-Wolf
  to the raw data. Verify resulting covariance is full-rank with condition
  number < 1e6 (while the sample covariance has condition number > 1e12).
- **Synthetic test**: Generate 64-channel Gaussian data with true rank 50
  (first 50 components have variance 1.0, last 14 have variance 0). Apply
  Ledoit-Wolf. Verify:
  - Shrinkage coefficient α ∈ [0, 1].
  - Result is positive definite (all eigenvalues > 0).
  - Condition number < 1e6.
- **Reference validation**: Compute Ledoit-Wolf on a fixed 10×10 dataset and
  compare α and the resulting covariance to scikit-learn
  `sklearn.covariance.LedoitWolf()` output (values match to 6 decimal places;
  embed reference values as test constants).
- **Identity limit**: For data where all channels are i.i.d. standard normal
  (n_samples >> n_channels), α should be close to 0 (sample covariance is
  already well-conditioned).
- **Edge cases**: n_samples < n_channels (severely rank-deficient, α should
  be close to 1), single sample (should fail gracefully), constant channel.
- **Coverage**: ≥75% line coverage for the Ledoit-Wolf implementation.

## TASK 16 — PythonRunner Utility

### Goal
Provide a standardized `PythonRunner` class in `src/libraries/utils/` for
launching Python scripts from C++ with real-time output streaming, progress
parsing, virtual-environment management, and prerequisite checking. This is the
foundation for **two** critical roles in mne-cpp:

1. **Production bridge**: ML training via `MLTrainer`, data-import scripts, and
   any future Python-delegated computation.
2. **Test cross-validation bridge**: Seamless invocation of MNE-Python from C++
   test frames to compute reference values on-the-fly, ensuring that MNE-CPP
   implementations produce equivalent results to MNE-Python — always validated
   against the live MNE-Python version, never against stale hard-coded constants.

A companion `PythonTestHelper` class provides test-frame convenience:
run MNE-Python code, parse JSON output, compare matrices, and gracefully fall
back to embedded reference values when Python is unavailable.

### Why
Several v2.2.0 features require Python for heavy computation (LSTM training,
model export to ONNX, potential data preprocessing). Equally important, every
algorithm that has an MNE-Python equivalent (forward solutions, inverse
operators, covariance estimation, source estimates, spectral analysis, cluster
statistics, …) needs numerical cross-validation. Today this is done manually by
pre-computing reference values in a Python script and pasting them into C++ test
constants — a fragile process that goes stale when either side changes.

`PythonRunner` + `PythonTestHelper` eliminate this problem:
- C++ test frames run MNE-Python snippets directly and compare the results
  against MNE-CPP output in the same test, with stated tolerances.
- When MNE-Python is not installed (e.g. a minimal CI job), the test falls back
  to embedded reference values but logs a clear warning.
- This keeps MNE-CPP permanently synchronized with MNE-Python without manual
  maintenance of reference data.

Additional concerns handled by `PythonRunner`:
- Python discovery and version checking
- Virtual-environment creation and dependency installation
- Real-time stdout/stderr streaming with line-level callbacks
- Progress protocol parsing (`[progress] 42.0%` lines)
- Timeout enforcement
- Security: input validation prevents command injection

```
    double scalarOrFallback(const QString& code, const QString& key,
                            double fallback) const;
    Eigen::MatrixXd matrixOrFallback(const QString& code, const QString& key,
                                      const Eigen::MatrixXd& fallback) const;

private:
    PythonRunner m_runner;
    mutable bool m_mnePythonChecked = false;
    mutable bool m_mnePythonAvailable = false;
};
```

### Progress protocol
Python scripts can report progress by printing lines matching:
```
[progress] 42.0%
[progress] 42.0% Epoch 21/50
```
`PythonRunner` parses these in real-time and forwards to the progress callback.
This is used by `MLTrainer` and the `mne_compute_cmne` CLI tool to display
training progress.

### Security
- `isPackageAvailable()` validates the package name against a strict regex
  (`^[A-Za-z_][A-Za-z0-9_.]*$`) to prevent injection.
- Script paths and arguments are passed as separate `QProcess` arguments (never
  concatenated into a shell command).
- Virtual-env creation uses `python3 -m venv`, not shell scripts.

### Acceptance criteria

**PythonRunner core:**
- **Python discovery**: `isPythonAvailable()` returns true when `python3` is on
  PATH; `pythonVersion()` returns a valid version string (e.g. "3.11.5").
- **Script execution**: `run("script.py", {"--arg", "value"})` executes the
  script and captures stdout/stderr. Exit code is correct.
- **Inline code**: `runCode("print('hello')")` captures "hello" in stdout.
- **Progress parsing**: Script printing `[progress] 75.0% Step 3/4` triggers
  the progress callback with pct=75.0 and message="Step 3/4".
- **Timeout**: `run()` with `timeoutMsec=500` on a script that sleeps for 5s
  returns `timedOut=true` and terminates the process.
- **Venv management**: `ensureVenv()` creates a venv if absent, `installDependencies()`
  installs from `pyproject.toml` or `requirements.txt`.
- **Package check**: `isPackageAvailable("os")` returns true;
  `isPackageAvailable("nonexistent_xyz")` returns false.
- **Injection prevention**: Package names with shell metacharacters
  (`"os; rm -rf /"`) are rejected by validation.
- **Edge cases**: Python not installed (isPythonAvailable returns false, run
  returns error), empty script path (error), script with non-zero exit code.

**PythonTestHelper cross-validation:**
- **MNE-Python detection**: `isMnePythonAvailable()` returns true when `mne`
  is importable, false otherwise. Result is cached.
- **JSON round-trip**: `runAndParseJson()` runs inline Python that prints a JSON
  dict, and returns it as `QJsonObject` with correct types.
- **Matrix comparison**: `compareMatrix()` runs MNE-Python code, extracts a 2D
  array from JSON, and compares element-wise against an Eigen matrix within
  `rtol`/`atol`. Returns true on match, logs max diff on failure.
- **Scalar comparison**: `compareScalar()` same for scalar values.
- **Fallback behavior**: When Python is unavailable, `scalarOrFallback()` and
  `matrixOrFallback()` return the provided fallback value (not an error),
  and log a warning indicating live cross-validation was skipped.
- **End-to-end demo test**: A test that computes a dSPM source estimate in
  MNE-CPP, then runs the equivalent in MNE-Python via `PythonTestHelper`,
  and verifies the peak value matches within `rtol=1e-5`. This test proves
  the full cross-validation workflow works.
- **No-Python CI**: The same test passes on a CI runner without MNE-Python
  installed, using the embedded fallback value instead.

**Coverage:** ≥75% line coverage for `python_runner.cpp` and
`python_test_helper.cpp`.

## TASK 17 — MNA/MNX Format Paper

### Goal
Write and submit a publication-ready paper describing the MNA/MNX analysis
container format and computational graph model. The target is an open-access
archive (arXiv, bioRxiv, or JOSS — Journal of Open Source Software) so the
format has a citable reference before the v2.2.0 release.

### Why
The MNA/MNX format introduces several novel concepts that don't exist in
neuroscience today (see comparison below). Publishing the design gives it:
1. A citable DOI for other projects to reference.
2. Peer review of the graph model and serialization design.
3. Visibility in the neuroimaging community, encouraging adoption.
4. Priority documentation — establishes MNE-CPP as the origin of the concept.

### Novelty — What MNA/MNX Does That Nothing Else Does

| Existing standard | Focus | Missing compared to MNA/MNX |
|---|---|---|
| **BIDS** | Directory naming convention for raw + derivative files | No computational graph, no parameters, no execution, no single-file container |
| **BIDS Derivatives** | JSON sidecars for processed data | Still just files + sidecars — no DAG, no typed dataflow, no re-execution |
| **NWB** | HDF5 container for neurophysiology data | Data storage only — no processing pipeline, no provenance graph |
| **NIDM** | W3C PROV-based provenance (RDF triples) | Records provenance but no execution engine, no typed ports, RDF-heavyweight; effectively dormant |
| **Nipype / Pydra** | Python DAG execution framework wrapping FSL/SPM/FreeSurfer/MNE | Has nodes + connections + execution, but: (a) Python-only, (b) runtime framework not a portable file format, (c) graphs are not self-contained serializable documents |
| **CWL** | Generic YAML workflow language | Describes tool invocations but no neuroscience type system, no incremental re-execution, no data embedding |
| **Boutiques** | JSON descriptors for CLI tool interfaces | Tool descriptions only, not pipeline graphs |
| **Neurodesk** | Containerized virtual desktop (Docker/Singularity) with 100+ pre-installed neuroimaging tools; deploy via browser, HPC, or cloud | Solves the *environment* problem (dependency hell, version conflicts) but not the *pipeline* problem — tools run in containers but processing steps, parameters, and dataflow remain implicit; no computational graph, no single-file pipeline description, no incremental re-execution; complementary to MNA/MNX (Neurodesk could *run* MNA pipelines inside its containers) |

MNA/MNX is the first format that is simultaneously:
- A **single portable file** (`.mna` JSON or `.mnx` CBOR binary)
- An **executable computational graph** with domain-typed ports (FiffRaw,
  Forward, SourceEstimate, FiffCov, RealTimeStream, IpcEndpoint, …)
- **Self-documenting** — every parameter on every node visible in plain text
- Both **serializable** (for sharing/archiving) and **executable** (for running)
- **Real-time capable** — stream ports turn the graph into an MNE Scan
  configuration, unifying offline analysis and live acquisition pipelines
- **Heterogeneous** — IPC nodes delegate to external tools (FreeSurfer, FSL,
  MRI scanners) within the same graph alongside native C++ and Python nodes
- **Adaptive** — the parameter tree with formula-driven bindings enables
  runtime parameter adjustment based on data characteristics, extending the
  concepts formalized in the Röll & Dinh MR patent (see bibliography)
- Roadmap to **packaging**: embed all referenced data in one `.mnx` binary

### Paper structure (suggested)

1. **Introduction** — The reproducibility crisis in neuroimaging pipelines;
   scattered files, invisible parameters, unreproducible analyses. Motivation
   from both offline research and real-time clinical/BCI use cases.
2. **Related work** — BIDS, NWB, NIDM, Nipype/Pydra, CWL, Boutiques,
   Neurodesk — what each solves and what it leaves open. The Röll & Dinh MR
   patent as prior art for adaptive parameter control in neuroimaging pipelines.
3. **The MNA/MNX format** — Dual serialization (JSON + CBOR), data hierarchy
   (Subject → Session → Recording → FileRef), schema versioning, `.mnx` magic
   header, packaging roadmap.
4. **Computational graph model** — ONNX-inspired DAG: `MnaNode`, `MnaPort`,
   `MnaDataKind` type system, `MnaOpSchema` + `MnaOpRegistry`, typed dataflow
   edges, graph-level inputs/outputs.
5. **Execution modes** — Batch execution (topological sort, validation,
   incremental re-execution via dirty propagation), real-time streaming
   (stream ports, ring buffers, MNE Scan integration), and IPC delegation
   (external tools, placeholder substitution, transport protocols).
6. **Parameter tree and adaptive bindings** — Hierarchical parameter store,
   formula-driven dynamic bindings, trigger conditions (on-change, periodic,
   manual), built-in operators. Relationship to the Röll & Dinh patent.
7. **Use case: M/EEG source imaging pipeline** — Walkthrough of a realistic
   graph (raw → covariance → forward → inverse → source estimate → statistics)
   showing parameter transparency and re-execution when one node changes.
8. **Use case: Real-time brain-computer interface** — A streaming graph that
   reads from an MEG acquisition via FIFF real-time protocol, applies online
   filtering and source localisation, and feeds a BCI classifier — all
   configured in a single `.mna` file loaded by MNE Scan.
9. **Implementation** — C++ library in MNE-CPP, Qt6 serialization, op
   registry extensibility, IPC node architecture, Python bridge for
   cross-validation.
10. **Discussion** — Limitations, future work (sub-graphs, loops/iteration,
    distributed execution, GUI graph editor, cloud execution), relationship
    to BIDS.

### CLI tool: `mne_inverse_pipeline`

**Location**: `src/applications/mne_inverse_pipeline/`

```
mne_inverse_pipeline \
  --raw   sub-01_ses-01_task-aud_raw.fif \
  --fwd   sub-01-fwd.fif \
  --cov   auto                          \
  --method dSPM                         \
  --lambda 0.1111                       \
  --out   sub-01-dspm-stc.fif           \
  --mna   sub-01-pipeline.mna           # also write the MNA project file
```

**Behaviour**:
1. Parses CLI args via `QCommandLineParser`.
2. Builds an `MnaGraph` programmatically (load nodes → cov → inv op → apply

3. Registers operation functions for each `opType` in `MnaOpRegistry`.
4. Executes the graph via `MnaGraphExecutor::execute()`.
5. Writes the source estimate to FIFF.
6. Serializes the complete `MnaProject` (including the graph with cached
   result paths) to the `.mna` JSON file.

This means the user gets **both** the result and a reproducible recipe.

### MNA operation registrations

The example registers these operations:

| `opType` | Library | Inputs | Outputs | Key attributes |
|----------|---------|--------|---------|----------------|

| `load_fiff_forward` | mne | `path: FilePath` | `forward: Forward` | — |
| `compute_covariance` | sts | `raw: FiffRaw` | `covariance: FiffCov` | `method`, `tmin`, `tmax` |
| `make_inverse_operator` | inv | `raw: FiffRaw`, `forward: Forward`, `covariance: FiffCov` | `inverse: Inverse` | `loose`, `depth` |
| `apply_inverse` | inv | `raw: FiffRaw`, `inverse: Inverse` | `stc: SourceEstimate` | `lambda2`, `method` (MNE/dSPM/sLORETA) |

### Acceptance criteria
- **CLI runs end-to-end**: `mne_inverse_pipeline --raw ... --fwd ... --out ...
  --mna ...` produces both a source estimate and a valid `.mna` file.
- **Graph validates**: The `.mna` file passes `MnaGraph::validate()` —
  acyclic, all ports typed, all connections resolved.
- **Graph re-executes**: Loading the `.mna` file and calling
  `MnaGraphExecutor::execute()` reproduces exactly the same source estimate.
- **Dirty propagation**: Changing `lambda2` in the `.mna` file and
  re-executing only re-runs the `apply_inverse` node (not covariance or
  forward).
- **Ops registered**: All 5 operation types are registered in
  `MnaOpRegistry` with proper `MnaOpSchema` (input/output ports, required
  attributes, descriptions).
- **Sample `.mna` file**: `resources/examples/mna_inverse_dspm.mna` is valid
  JSON, passes schema validation, and runs against the MNE sample dataset.
- **Cross-validated**: Source estimate output matches MNE-Python's
  `mne.minimum_norm.apply_inverse()` within tolerance (via PythonTestHelper).

## Artifact Naming Convention

All release artifacts follow a **unified naming scheme** — no `v` prefix on
version numbers.  Git tags remain `v2.2.0` (standard convention), but artifact
filenames use the bare version `2.2.0`.

### Pattern

```
mne-cpp-{version}-{platform}-{linkage}-{arch}.{ext}        # binary release
mne-cpp-sdk-{version}-{platform}-{arch}.{ext}               # SDK (no linkage — headers + matched libs)
mne-cpp-{version}-{platform}-installer-{arch}.{ext}         # IFW installer
mne-cpp-{version}-wasm.tar.gz                                # WebAssembly
```

### Concrete examples (v2.2.0 tag)

| Artifact | Filename |
|----------|----------|
| Linux dynamic binary | `mne-cpp-2.2.0-linux-dynamic-x86_64.tar.gz` |
| Linux static binary | `mne-cpp-2.2.0-linux-static-x86_64.tar.gz` |
| macOS dynamic binary | `mne-cpp-2.2.0-macos-dynamic-arm64.tar.gz` |
| Windows dynamic binary | `mne-cpp-2.2.0-windows-dynamic-x86_64.zip` |
| Linux SDK | `mne-cpp-sdk-2.2.0-linux-x86_64.tar.gz` |
| macOS SDK | `mne-cpp-sdk-2.2.0-macos-arm64.tar.gz` |
| Windows SDK | `mne-cpp-sdk-2.2.0-windows-x86_64.zip` |
| Linux installer | `mne-cpp-2.2.0-linux-installer-x86_64.run` |
| macOS installer | `mne-cpp-2.2.0-macos-installer-arm64.dmg` |
| Windows installer | `mne-cpp-2.2.0-windows-installer-x86_64.exe` |
| WebAssembly | `mne-cpp-2.2.0-wasm.tar.gz` |

### Special channels

| Channel | Prefix | Example |
|---------|--------|---------|
| `main` branch (latest) | `mne-cpp-latest` | `mne-cpp-latest-linux-dynamic-x86_64.tar.gz` |
| `staging` branch (dev) | `mne-cpp-dev` | `mne-cpp-dev-wasm.tar.gz` |

### Implementation

The `v` is stripped in each workflow's "Determine release info" step:

```bash
TAG="${GITHUB_REF#refs/tags/}"
echo "prefix=mne-cpp-${TAG#v}" >> "$GITHUB_OUTPUT"
```

The git tag (`v2.2.0`) is preserved for `gh release upload` targets; only the
artifact filename drops the `v`.
